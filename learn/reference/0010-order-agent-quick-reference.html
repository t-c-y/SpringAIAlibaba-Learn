API 速查表

API / 概念一句话说明
X-User-Id HTTP 头Controller 入口从 header 取 userId
UserContext ThreadLocalController set → ChatClient/Tool get → finally clear
HITL (Human-in-the-loop)敏感操作（退款、下单）必须用户二次确认
defaultTools(OrderTools.class)工具内 UserContext.currentUserId() 获取当前用户
幂等键订单操作要传或生成 idempotency key 防重复

典型代码模板

@PostMapping("/order/ask")
public Object ask(@RequestHeader("X-User-Id") String uid,
                 @RequestBody AskRequest req) {
  try { UserContext.set(uid);
    return chatClient.prompt().user(req.question())
        .tools(orderTools).call().content();
  } finally { UserContext.clear(); }
}

踩坑清单

  • 绝对不能让模型传 userId 参数（模型会撒谎、会被注入编造成任意用户）
  • ThreadLocal 必须在 finally 里 clear()，否则线程池复用导致 A 用户看到 B 用户数据
  • 退款/下单等敏感操作要先返回"确认消息"等用户下一轮明确确认再执行（HITL）
  • 工具内必须二次校验订单归属（即使 ThreadLocal 有 uid 也要再校验 order.getOwnerId()）
  • 订单操作必须幂等（用 idempotency key 或订单状态机防重复）
  • 不要在工具里返回订单号明文给日志要脱敏
