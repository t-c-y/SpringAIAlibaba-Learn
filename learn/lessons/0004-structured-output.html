Lesson 0004 · Spring AI Alibaba

结构化 JSON 输出：告别"手动 parse LLM 文本"

本课目标：会用 .entity(Class) 让模型返回可直接反序列化的 Java 对象或集合。

配套 Demo：demos/05-structured-output。参考卡片：结构化输出快速参考。

你今天要获得的一个能力

能力目标：面对"把 LLM 自由文本转成业务可用对象"的需求，能用 ChatClient.entity() + Java record 一次性搞定 schema 注入、模型约束、反序列化；并能解释 BeanOutputConverter 内部链路，定位"JSON 解析失败"的三类典型问题。

你在写 Java 业务接口时最怕的不是调用模型，而是模型返回的那段文本不稳定：有时带解释、有时带 ```json 包裹、字段名会漂移。本节课解决的就是这件事——让模型"必须"按你声明的类型说话，且结果直接落到 Java 对象里。

原理：BeanOutputConverter 如何让模型返回 JSON

很多同学以为 .entity(Class) 是"拿到字符串后用 Jackson 反序列化"，这只说对了一半。真正的关键在于——在发给模型之前，Spring AI 已经把你 Java 类型翻译成了 JSON Schema，并塞进了 System Prompt，模型是"照着 Schema 填写"的，不是自由发挥后再硬 parse。

┌─────────────────────────────────────────────────────────────────┐
│ 1. 你声明 record LessonOutline(String title, String summary …) │
└────────────────────────────────┬────────────────────────────────┘
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. BeanOutputConverter 反射 record → 生成 JSON Schema            │
│    (字段名/类型/必填/description 都在 Schema 里)                  │
└────────────────────────────────┬────────────────────────────────┘
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. Schema 作为特殊指令拼进 System Prompt:                        │
│    "Your response should be in JSON format …  schema: {…}"     │
└────────────────────────────────┬────────────────────────────────┘
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. 模型(qwen-plus / qwen3.7-plus 等)按 Schema 生成 JSON 文本       │
│    (支持 structured output 的模型会严格遵守 key/类型)             │
└────────────────────────────────┬────────────────────────────────┘
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│ 5. BeanOutputConverter 从响应文本里抽取 JSON → Jackson 反序列化   │
│    直接落到 LessonOutline / List<Checkpoint> 实例                  │
└─────────────────────────────────────────────────────────────────┘

为什么要把 Schema 注入 Prompt？因为大模型本质上是在"下一个 token 预测"，只有当它在生成的每一步都"看到"Schema 约束，才不会在输出中凭空加字段、换字段名、混入自然语言解释。支持"结构化输出 / JSON Mode"的模型（如 Qwen-Plus、Qwen3 系列、GPT-4o 等）会在服务端再做一次强约束，效果更好；不支持的模型则仅靠 Prompt 引导，稳定性差一些。

这也解释了两件事：(1) 字段名和类型必须在 Java 侧写清楚——它们就是 Schema 的来源；(2) temperature 必须低——温度高时采样空间大，模型更容易"改写"key 名或加多余字段，即便有 Schema 约束也会漏。百炼平台的 qwen3.7-plus、qwen-plus 对 JSON Schema 支持度较好，结构化输出稳定性是经过生产验证的。

核心 API 与 record 写法

先把两个 record 写对——字段名、类型、@JsonPropertyDescription 注解都会直接映射到 JSON Schema 的 properties 与 description，从而影响模型输出质量。

import com.fasterxml.jackson.annotation.JsonPropertyDescription;
import java.util.List;

/** 课程大纲：嵌套一个 Checkpoint 列表 */
public record LessonOutline(
    @JsonPropertyDescription("课程标题，不超过 20 字")
    String title,

    @JsonPropertyDescription("一段中文摘要，80-120 字，说明本课讲什么、学完能做什么")
    String summary,

    @JsonPropertyDescription("本课分几节，每节一个简短标题")
    List<String> sections,

    @JsonPropertyDescription("难度等级：beginner / intermediate / advanced 三者之一")
    String difficulty
) {}

/** 检查点：用于 List<Checkpoint> 场景 */
public record Checkpoint(
    @JsonPropertyDescription("检查点序号，从 1 开始")
    Integer id,

    @JsonPropertyDescription("一句话描述学员应该已经掌握什么")
    String description,

    @JsonPropertyDescription("学员自检方式：提问/命令/代码片段任选其一")
    String verifyMethod
) {}

Controller 里的调用非常薄：

@RestController
@RequestMapping("/struct")
@RequiredArgsConstructor
public class StructController {

    private final ChatClient chatClient;

    /** 返回单个对象 */
    @PostMapping("/lesson")
    public LessonOutline lesson(@RequestBody TopicRequest req) {
        return chatClient.prompt()
            .user(u -> u.text("请围绕主题生成一份课程大纲，主题：{topic}")
                        .param("topic", req.topic()))
            // temperature 设低，是结构化输出的配套；也可在 defaultOptions 全局设
            .options(DashScopeChatOptions.builder()
                    .withTemperature(0.1f)
                    .build())
            .call()
            .entity(LessonOutline.class);
    }

    /** 返回集合必须用 ParameterizedTypeReference 保留泛型 */
    @PostMapping("/checkpoints")
    public List<Checkpoint> checkpoints(@RequestBody TopicRequest req) {
        return chatClient.prompt()
            .user(u -> u.text("围绕主题生成 5 条学习检查点，主题：{topic}")
                        .param("topic", req.topic()))
            .call()
            .entity(new ParameterizedTypeReference<>() {});
    }

    public record TopicRequest(String topic) {}
}

注意：ParameterizedTypeReference 是 Spring 核心里就有的泛型令牌（org.springframework.core.ParameterizedTypeReference），和 RestTemplate.exchange 里保留泛型是同一个套路——Java 类型擦除后 List 的泛型信息靠它传下来，才能生成正确的"数组 of Checkpoint"Schema。

什么时候用 Class、什么时候用 ParameterizedTypeReference、什么时候嵌套

三种返回结构对应三种写法，别搞混：

想要的返回写法Schema 形态常见坑
单个对象 LessonOutline.entity(LessonOutline.class){type:object, properties:{…}}字段必须是具体类型，不能是 Object
同类型集合 List<Checkpoint>.entity(new ParameterizedTypeReference<List<Checkpoint>>(){}){type:array, items:{…Checkpoint…}}漏写泛型令牌会导致 Jackson 反序列化成 List<LinkedHashMap>
嵌套对象（如 LessonOutline 内含 List<Checkpoint>）字段直接声明成 record/List 类型，仍传 .entity(LessonOutline.class)Schema 里该字段是 {type:array, items:{…Checkpoint…}}内层 record 也要有 @JsonPropertyDescription，否则模型对内层字段描述不足
异构集合（如"可能是 A 也可能是 B"）用一个包装 record + 一个 String type 判别字段（示意）oneOf/anyOf（需 Spring AI 版本支持）Spring AI 当前版本对 oneOf 支持有限，建议用"type 判别字段 + 业务分支"替代

关键参数与配置

temperature（温度）

建议值：0.0 ~ 0.1，默认按全局 ChatOptions 通常是 0.7 或更高。结构化输出必须调低，否则 key 名漂移率显著上升。

model（模型）

建议：qwen-plus / qwen3-plus 系列。这类模型在服务端支持 JSON 模式或结构化输出约束，不建议在结构化输出场景使用不支持 JSON Mode 的弱模型。

@JsonPropertyDescription

强烈建议每个字段都加。它会变成 Schema 里字段的 description，是模型理解"这个字段该填什么"的唯一中文提示——比你在 user prompt 里重复写更稳定。

maxTokens

嵌套层级多、字段多时适当调大（如 1024 → 2048），否则输出被截断会导致 JSON 不完整、Jackson 直接抛 JsonParseException。

这些参数既可在 application.yml 的 spring.ai.dashscope.chat.options.* 做全局默认，也可在单次调用 .options(DashScopeChatOptions.builder()…) 覆盖。结构化输出接口建议全局默认 temperature=0.1，避免每次都写。

边界与常见坑

坑 1：模型输出被 ```json … ``` 包裹，直接 JsonParseException。

典型报错：com.fasterxml.jackson.core.JsonParseException: Unexpected character ('`' (code 96)): Expected space separating root-level values。原因：模型没严格遵守"只输出 JSON"指令，加上了 markdown 代码块。解决：在 System Prompt 中显式追加"不要使用 markdown 代码块包裹，不要输出任何解释性文字，直接输出 JSON"；或升级到支持 JSON Mode 的模型（如 qwen-plus）在服务端做硬约束。BeanOutputConverter 本身对简单的 ```json 前缀有一定容错，但不要依赖它。

坑 2：字段缺失 / 类型错误，Jackson 抛 MismatchedInputException。

典型报错：Cannot deserialize value of type `int` from String "N/A" 或某个字段为 null而你业务没判空。原因两类：(a) temperature 偏高、模型在自由发挥；(b) record 字段类型太"严"但 description 没写清（比如模型把"id"输出成"第1条"）。解决：temperature 降到 0.1 以下；字段用包装类型（Integer 而非 int）以便处理缺省；为每个字段补 @JsonPropertyDescription 指明类型与取值范围。

坑 3：幻觉字段——模型输出了你没声明过的 key。

现象：JSON 里多出 "extraComment": "…"、"source": "…" 之类的字段，Jackson 默认会忽略（不报错），但下游逻辑如果遍历 key 可能踩坑。原因：user prompt 里没明确"只输出这些字段，不要额外加信息"，模型为了"有帮助"会主动加解释字段。解决：在 System Prompt 明确"必须严格按 schema 返回，禁止添加任何未声明字段"；同时在 Jackson 侧配置 DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES 为 false（Spring AI 默认行为），或者在关键业务上主动打开该开关以便及早发现漂移。

坑 4（Bonus）：List 裸传 Class 丢泛型。

.entity(List.class) 会得到 List<LinkedHashMap>，强转 List<Checkpoint> 会在运行时 ClassCastException。务必用 new ParameterizedTypeReference<List<Checkpoint>>() {}。

Java 侧配合要点

  • record 字段名 = JSON 字段名。
  • 字段类型用具体类型（int / String / List<String>），别用 Object。
  • 温度务必调低（0.1 及以下），避免模型自由发挥破坏 JSON。

反例：不用 .entity()

Demo 还提供 /struct/raw，让你看看模型返回可能包裹 ```json、加解释、字段名不一致等麻烦。生产代码里不要这样干。
