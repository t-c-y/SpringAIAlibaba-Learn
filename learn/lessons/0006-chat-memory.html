Lesson 0006 · Spring AI Alibaba

多轮对话 Memory：会话上下文的第一课

本课目标：会用 ChatMemory + MessageChatMemoryAdvisor 给 ChatBot 装上记忆，通过 conversationId 隔离用户。

配套 Demo：demos/07-chat-memory。参考卡：Memory 快速参考。

你今天要获得的一个能力

能力目标：能够在 Spring Boot 里用 ChatMemory + MessageChatMemoryAdvisor 给 ChatClient 装上多轮记忆，并明确 conversationId、窗口大小、存储后端三者在生产环境下的边界。

大模型本身是无状态的——你每一次 call() 它都像第一次见到你。"记忆"不是模型自带的能力，而是应用层在请求前把历史拼进 Prompt、请求后把新对话写回存储。这节课就把这件事的完整链路讲清楚。

三个组件

记忆能力由三件套组成：ChatMemory（存储接口）+ MessageChatMemoryAdvisor（在 advisor 链里负责读/写）+ conversationId（请求级参数，用来区分会话）。

// ① 存储：Spring AI 1.1+ 推荐 MessageWindowChatMemory（滑窗裁剪）
ChatMemory memory = MessageWindowChatMemory.builder()
        .maxMessages(20)           // 保留最近 20 条消息
        .build();

// ② 把 Advisor 挂到 ChatClient 上（全局一次注册即可）
ChatClient client = builder.defaultAdvisors(
        MessageChatMemoryAdvisor.builder(memory).build()
).build();

// ③ 每次请求必须传入 conversationId 做会话隔离
client.prompt().user(q).advisors(a -> a
        .param("chat_memory_conversation_id", cid)
).call().content();

旧版 Spring AI 里常写 new InMemoryChatMemory() 以及通过 CHAT_MEMORY_RETRIEVE_SIZE_KEY 控制取回条数；Spring AI 1.1+ 这两件事合并到 MessageWindowChatMemory.maxMessages，advisor param 只需要传 chat_memory_conversation_id。

ChatMemory 工作原理：让无状态的 LLM 看起来有记忆

LLM 本身没有"会话"的概念——每次请求发给模型的，只有这一次 Prompt 里出现过的文本。所以"记住前文"的真实做法，是在调用模型之前，由你自己把该会话的历史消息读出来、拼到本次 Prompt 里；模型返回后，再把这一轮新的 user/assistant 消息写回存储。Spring AI 把这两步封装在 MessageChatMemoryAdvisor 里，挂在 ChatClient 的 advisor 链上，你不用手写拼接逻辑。

用户发新问题
   │
   ▼
┌─────────────────────────────────────────────┐
│ ChatClient.prompt().user(q).call()          │
│   │                                         │
│   ▼ ① 请求前 (before)                       │
│ MessageChatMemoryAdvisor:                   │
│   memory.get(conversationId) → 历史消息     │
│   把历史 + system + 本次 user 组装成 Prompt │
│   │                                         │
│   ▼ ②                                       │
│ DashScope ChatModel → 大模型                │
│   │                                         │
│   ▼ ③ 请求后 (after)                        │
│ MessageChatMemoryAdvisor:                   │
│   memory.add(conversationId, userMsg)       │
│   memory.add(conversationId, assistantMsg)  │
└─────────────────────────────────────────────┘
   │
   ▼
返回 assistant 文本给前端

关键点是 advisor 在请求前后各执行一次：before 负责"回忆"（从存储拉历史塞进 Prompt），after 负责"记下来"（把新 exchange 写回）。模型全程都不知道自己被这样对待，它每次看到的只是一段更长的 Prompt。

Spring AI 的 advisor 是责任链模式，除了 MemoryAdvisor，你后面还会见到 QuestionAnswerAdvisor（RAG 检索）、ToolCallingAdvisor（工具调用）等。MemoryAdvisor 通常放在链的前面，因为后续 advisor 追加的消息也需要被写入历史。

核心 API 与完整代码

下面是 demos/07-chat-memory 真实可运行的最小配置（Spring AI 1.1+，使用 MessageWindowChatMemory）：

@Configuration
public class ChatMemoryConfig {

    // ① 定义 ChatMemory：滑动窗口，默认保留最近 20 条消息
    @Bean
    public ChatMemory chatMemory() {
        return MessageWindowChatMemory.builder()
                // .maxMessages(20)     // 窗口大小，默认 20
                // .chatMemoryRepository(...)  // 生产换成 Redis/JDBC 实现
                .build();
    }

    // ② 把 MessageChatMemoryAdvisor 注册为默认 advisor
    @Bean
    public ChatClient memoryChatClient(ChatClient.Builder builder,
                                       ChatMemory chatMemory) {
        return builder
                .defaultSystem("你是一名企业级 Java 后端 + Spring AI Alibaba 学习助手……")
                .defaultAdvisors(
                        MessageChatMemoryAdvisor.builder(chatMemory).build()
                )
                .build();
    }
}

Controller 里每次请求必须传入 chat_memory_conversation_id，否则所有请求会进入同一个 default 会话：

@PostMapping("/chat/ask")
public Object ask(@RequestBody AskRequest req) {
    String cid = req.conversationId();   // 生产必须从登录态派生，不能信前端
    String answer = chatClient.prompt()
            .user(req.question())
            .advisors(a -> a.param("chat_memory_conversation_id", cid))
            .call()
            .content();
    return Map.of("success", true, "answer", answer, "conversationId", cid);
}

常量 key "chat_memory_conversation_id" 对应 AbstractChatMemoryAdvisor.CHAT_MEMORY_CONVERSATION_ID_KEY；旧版本还需要传 CHAT_MEMORY_RETRIEVE_SIZE_KEY，Spring AI 1.1+ 窗口大小由 MessageWindowChatMemory.maxMessages 统一控制，不再走 advisor param。

ChatMemory 实现对比

Spring AI 内置了几种 ChatMemory 实现，选择哪一种决定了"记忆"存在哪、保留多久、能否跨实例共享。

InMemoryChatMemoryRepository / InMemoryChatMemory

最朴素实现：一个 ConcurrentHashMap<conversationId, List<Message>>。重启即丢失，不支持多实例共享。仅适用于本地 Demo、单用户测试。

MessageWindowChatMemory（推荐默认）

基于任意 ChatMemoryRepository（默认 InMemory），但在写入时自动裁剪到最近 K 条消息。解决"上下文无限增长"的问题。Demo 和大多数单实例场景够用。

JdbcChatMemoryRepository / RedisChatMemoryRepository（示意）

Spring AI 扩展包提供的持久化后端。重启不丢、可多实例共享，是生产环境的首选。需要额外引入对应 starter 并配置表结构/Redis 连接。

关键参数与配置

参数 / 方法默认值何时调整
MessageWindowChatMemory.maxMessages20业务需要长上下文时调大（如客服排查多轮问题），但注意 token 成本与延迟；短问答场景可降到 10。
chat_memory_conversation_id"default"每次请求都要传。值必须由服务端从登录态派生（如 userId + sessionId），绝不能信任前端自报。
defaultAdvisors(MessageChatMemoryAdvisor...)无在 ChatClient 构造时统一注册，业务代码不必每次 .advisors(...) 重复加；但 conversationId 仍需每次请求传。
ChatMemory.get(cid)—用于调试/展示历史；Spring AI 1.1+ 只接受 cid，条数由窗口控制。
ChatMemory.clear(cid)—用户主动"清空记忆"、会话结束、登出时调用。

窗口大小 K 的 trade-off：K 越大，模型越能回忆前文，但每次请求携带的历史消息越多，token 消耗与延迟线性上升；K 过小则模型会在 5~10 轮之后"忘掉"你一开始说过的名字、目标。企业落地通常取 10~30，并结合摘要（summarization）或 RAG 处理更长的会话。

会话隔离

不传 conversationId 就默认合到 default 会话，用户之间会串数据。 生产必须由服务端根据用户 + 会话槽生成。

边界与常见坑

  1. 用 InMemory 上生产：InMemoryChatMemory/默认的 MessageWindowChatMemory 都是进程内存储，应用重启记忆全丢，多实例部署下同一个用户的请求打到不同实例会"失忆"或"串记忆"。生产必须换成 JDBC 或 Redis 后端。
  2. conversationId 信任前端：如果 Controller 直接拿请求里的 conversationId 去查历史，恶意用户可以遍历别人的 id 偷看到对话内容。正确做法是从登录态（如 JWT/Session）解析出 userId，再结合前端传的会话槽号组合成服务端可控的 cid。
  3. 不限制记忆长度：没有窗口或不做摘要，历史会随轮次线性增长，最终触发模型上下文长度上限或产生巨额 token 费用。务必设置 maxMessages，长会话场景叠加摘要记忆（SummaryChatMemory）。
  4. 敏感信息长期留存：ChatMemory 会把用户输入原样存下，如果里面包含身份证号、手机号、病历等 PII，就必须评估 TTL、加密与合规（如 GDPR/个保法）。对敏感字段在写入前做脱敏，或在会话结束后立即 clear。
