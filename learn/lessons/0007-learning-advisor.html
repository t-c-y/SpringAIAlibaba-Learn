本课目标：在多轮记忆基础上叠加"用户画像"这一层长期记忆，理解两级记忆的分工。

配套 Demo：demos/08-learning-advisor。参考卡：学习顾问快速参考。

你今天要获得的一个能力：能像组装 Spring MVC Interceptor 链一样，用 Advisor 在 ChatClient 调用前后注入"会话记忆 + 用户画像"两类上下文，并说清它们在链中的顺序、参数如何传递、生产要替换哪些组件。

Advisor 拦截器链：ChatClient 的中间件模型

如果你熟悉 Spring MVC 的 HandlerInterceptor 或 Servlet 的 Filter，Spring AI 的 Advisor 就是同一套思路：在一次模型调用前后插入可复用的横切逻辑。记忆注入、日志、重试、限流、画像、RAG 检索——全都是 Advisor。

两种 Advisor，两种调用路径

CallAdvisor

同步 call() 路径使用。实现 advisedCall()，拿到完整 ChatResponse 后可以改写或落库。
MessageChatMemoryAdvisor、QuestionAnswerAdvisor（RAG）都实现它。

StreamAdvisor

流式 stream() 路径使用。实现 advisedStream()，拿到 Flux<ChatResponse>，必须在流结束钩子（doOnComplete/doOnTerminate）里保存记忆，否则会漏写。

很多官方 Advisor（如 MessageChatMemoryAdvisor）同时实现两个接口，所以 call/stream 都能用；自定义 Advisor 如果只实现其中一个，切到另一种调用方式时就"不生效"，这是新手最常见的第一个坑。

两层记忆

短期 / 会话

  • ChatMemory 存储
  • conversationId 隔离
  • "这次会话说过的话"

长期 / 画像

  • LearnerProfileStore 存储
  • userId 隔离
  • "稳定的用户属性"

顺序为什么重要：先画像，再记忆？

两级 Advisor 的顺序直接影响 System Prompt 的最终形态。Demo 里注册顺序是 ProfileAdvisor → MessageChatMemoryAdvisor，背后有一个刻意的设计：

  1. 画像 Advisor 在前（order 较小）：在 before 阶段向 prompt 追加 System Message，描述"你服务的学员是谁"。这是身份设定，应该最先就位。
  2. 记忆 Advisor 在后（order 较大）：从 ChatMemory 取出最近 N 轮消息追加到 prompt 末尾。它只追加消息，不会改写已有 System Message，不会覆盖画像。
  3. after 阶段顺序反转：记忆 Advisor 必须在响应完成后把本轮 Q/A 写入 ChatMemory；画像 Advisor 一般不做 after 操作（画像不应被模型自动修改）。

顺序反了会怎样？如果先注入历史消息、后注入画像 System Message，MessageChatMemoryAdvisor 内部会在第一次调用时把初始 System Prompt 也存进 memory，后续轮次 System Message 被重复拼接；更糟的情况是某些 Advisor 使用 promptContents.add(0, ...) 插到头部，导致你手写的 system 模板被覆盖。结论：画像/系统设定类 Advisor 始终 order 最小；记忆/检索类 Advisor 紧随其后；审计/日志类 Advisor 放最后。

怎么把画像给模型

不是每轮都塞 user 消息，而是渲染进 System Prompt。画像不是对话消息，它是"用户属性"，不该出现在历史消息流里——这正是我们不把画像写进 ChatMemory 的原因。

String system = """
你是学习顾问，服务对象画像：

回答规则：...
""".formatted(profile.toSystemFragment());

核心 API：注册 Advisor 与传参

ChatClient 提供两个层级注册 Advisor：

  • defaultAdvisors(...)：在构建 Client 时全局注册，所有请求都走；适合记忆、日志、RAG 这类通用横切逻辑。
  • advisors(...)：在单次 prompt() 请求上临时追加；适合本次请求特有的画像、实验性开关。

Advisor 之间通过 AdvisedRequest.adviseContext()（一个 Map<String,Object>）传参，对外暴露为 .param(k, v)。你看到的 chat_memory_conversation_id、chat_memory_response_size 本质都是 Advisor 读取的 param。

// 1) 全局注册：记忆 Advisor + 画像 Advisor
ChatClient client = ChatClient.builder(chatModel)
    .defaultSystem(systemTemplate)
    .defaultAdvisors(
        MessageChatMemoryAdvisor.builder(chatMemory).build(),   // 短期记忆
        new ProfileAdvisor(profileStore)                       // 自定义画像 Advisor
    )
    .build();

// 2) 请求级别通过 param() 把 userId / conversationId 塞给 Advisor
ChatResponse resp = client.prompt()
    .user(question)
    .param("userId", userId)                         // ProfileAdvisor 读取它
    .param("chat_memory_conversation_id", convId)    // MessageChatMemoryAdvisor 读取它
    .param("chat_memory_response_size", 20)          // 保留最近 20 条消息
    .call()
    .chatResponse();

// 3) 自定义 Advisor 里如何取 param
public class ProfileAdvisor implements CallAdvisor, StreamAdvisor {
    @Override
    public AdvisedRequest adviseRequest(AdvisedRequest req, Map<String,Object> ctx) {
        String userId = (String) req.adviseContext().get("userId"); // 从 param 取
        LearnerProfile p = store.load(userId);
        // 在 System Message 头部追加画像片段
        req.promptContents().add(0, new SystemMessage(p.toSystemFragment()));
        return req;
    }
    // after 阶段：通常不修改画像，直接返回 response
}

adviseContext() 在同一条链上共享——所以上游 Advisor 塞的值下游能读到，这是链上协作的核心机制，但也意味着命名要 namespace 化（例如 profile.userId 而不是 userId），避免和官方 Advisor 的 key 冲突。

双层记忆的设计取舍

短期 ChatMemory

  • 存"对话消息"（User/Assistant/System）
  • 按 conversationId 隔离，可被滑动窗口裁剪
  • 会被 MessageChatMemoryAdvisor 在 after 阶段自动追加新消息
  • 面向"本次会话连贯性"

长期 LearnerProfileStore

  • 存"用户属性"（岗位、已掌握技能、学习预算）
  • 按 userId 隔离，跨会话累积
  • 不会被 Advisor 自动写入；必须走专用 API（或 Tool Calling 经确认后写入）
  • 面向"个性化服务质量"

为什么不把画像也塞进 ChatMemory？因为 ChatMemory 的写入路径是"每轮 after 自动追加"，这意味着模型的输出会被自动写入。而画像属于"事实/偏好数据"，必须受业务规则保护——模型不能因为一轮对话里用户随口说"我不想学了"就自动把目标岗位清空。把两类数据拆开，写路径分开治理，是把 LLM 应用做稳的关键。

边界与常见坑（生产踩过）

  1. Map 存储不能上生产。 Demo 里 LearnerProfileStore 是 ConcurrentHashMap，InMemoryChatMemory 也是内存，重启就丢、多实例不共享、还没 TTL。生产替换为 JdbcChatMemory（Spring AI 官方）或基于 Redis/数据库自己实现 ChatMemoryRepository / LearnerProfileStore。
  2. userId 不能信任前端。 /advisor/ask 里 param("userId", ...) 的值必须来自登录态（JWT / Session），而不是前端传什么就是什么。否则用户 A 传 userId=B 就能读到 B 的画像和历史——这是真实安全事故。
  3. Advisor order 错乱会覆盖 System Prompt。 多个 Advisor 都在操作 System Message 时，顺序错一条，你精心写的 system 模板可能被 add(0, ...) 插到后面，模型就"人格错乱"。显式实现 Ordered 接口或用 @Order 注解，把系统设定类 Advisor 的 order 设成最小（如 Ordered.HIGHEST_PRECEDENCE）。
  4. ChatMemory 并发写要加锁 / 串行化。 MessageChatMemoryAdvisor 的 after 阶段会 memory.add(conversationId, ...)，同一 conversationId 并发两请求（用户连发、SSE 重连、重试）会把历史顺序写乱。生产上要按 conversationId 加分布式锁或对同一 session 的请求串行化。
  5. 流式接口别忘实现 StreamAdvisor。 只实现 CallAdvisor 的自定义 Advisor，切到 stream() 时完全不生效，还不报错——排查时特别容易绕进去。
  6. param key 命名冲突。 chat_memory_conversation_id、chat_memory_response_size 是官方保留 key；自定义参数建议加业务前缀（profile.userId、biz.tenantId），避免升级版本后被官方占用。

课后问答

  • "画像字段太多，会不会 System Prompt 太长撑爆 context？" — 会。解决思路：按场景做字段裁剪（教学答疑场景不要塞购物偏好），或对画像本身做"摘要压缩"，保留最影响回答的 3-5 个字段。
  • "能不能把画像存向量库，让模型自己检索？" — 不推荐。画像属于强约束属性，不是"可能相关的参考资料"，RAG 检索有召回率问题，模型可能忽略。画像应该像 system prompt 一样确定注入，而不是概率性召回。
  • "如果要让模型根据对话自动更新画像（比如发现用户换了目标岗位），怎么设计才安全？" — 不要让 Advisor 在 after 阶段直接写画像。用 Tool Calling 暴露一个 updateProfile(field, newValue, reason) 工具，由模型主动调用；后端做字段白名单 + 置信度/人工确认 + 变更审计，避免模型"幻觉"污染长期数据。
