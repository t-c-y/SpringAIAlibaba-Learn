Lesson 0014 · Spring AI Alibaba

智能客服 Agent：Memory + RAG + Tool 三合一

本课目标：把前面学的三大能力合到一个 Agent 上，理解 Advisor 顺序和"读工具/写工具"的分层。

配套 Demo：demos/15-customer-service-agent。参考卡：综合 Agent 快速参考。

你今天要获得的一个能力：搭建一个把会话记忆（Memory）、知识检索（RAG）、工具调用（Tool）三者编排在一起的客服 Agent。这是你接触的第一个"真正意义上的 Agent"——它不再只靠一个能力回答，而是先读历史、再查政策、再调业务接口，最后才给用户答复。上完这节课，你能说清 Advisor 链的顺序为什么不能乱、用户身份如何安全注入、以及哪些动作必须走人工审批。

数据流

用户消息
  ↓ MessageChatMemoryAdvisor 拼历史
  ↓ QuestionAnswerAdvisor 检索知识库
  ↓ LLM 决策
  ↓ Tool (查订单 / 提交退款)
  ↓ 生成回答
  ↓ 记忆写回

原理机制：Advisor 链顺序为什么重要

Spring AI 的 Advisor 是一层层包裹 ChatClient.call() 的拦截器，和 Servlet Filter 的思路完全一致——谁先注册谁先处理入站消息、谁最后处理出站响应。在客服 Agent 里，顺序不是"好看"的问题，顺序错了功能就错。

                入站方向 (before)                        出站方向 (after)
   ┌──────────────────────────────────────┐            ┌───────────────────┐
   │  ① MessageChatMemoryAdvisor         │            │  写回记忆          │
   │     - 把历史会话拼进 Prompt           │            │  - 保存本轮问答     │
   ├──────────────────────────────────────┤            ├───────────────────┤
   │  ② QuestionAnswerAdvisor (RAG)      │            │  (无出站动作)      │
   │     - 用"当前问题+历史"检索政策       │            │                   │
   │     - 把召回段落注入 System Prompt   │            │                   │
   ├──────────────────────────────────────┤            ├───────────────────┤
   │  ③ UserContextAdvisor (身份)        │            │  (无出站动作)      │
   │     - 从 ThreadLocal 取 userId       │            │                   │
   │     - 注入 "当前用户：u001"          │            │                   │
   ├──────────────────────────────────────┤            ├───────────────────┤
   │  ④ 模型 + ToolCallback (LLM 决策)   │──────────→ │  ④' Tool 返回结果  │
   │     - 模型看完整 Prompt 决定          │            │      交回模型重写   │
   │       要不要调 CustomerServiceTools  │            │      最终自然语言   │
   └──────────────────────────────────────┘            └───────────────────┘

为什么 Memory 必须在最前面。RAG 做向量检索时需要"完整语义"——用户上一轮问"我买的东西发货了吗"，这一轮只说"那能退吗"，没有历史就会把"退"检索到退货政策而不是针对订单的退款规则。先注入历史，后续的 RAG 才能用"上下文问题"去检索。

为什么 RAG 在 Tool 之前。模型决定调不调工具，依赖的是已经拼好政策知识的 Prompt。如果 Tool 先执行（例如先退款了），RAG 才告诉模型"该商品签收超过 7 天不能退"，那已经晚了——动作已经发生。正确顺序是：模型看到政策后再决定调什么工具、传什么参数。

为什么 UserContext 要在 RAG 和 Tool 之间（或更早）。Tool 里的 listMyOrders()、requestRefund() 都需要当前 userId；这个值必须由服务端从 X-User-Id Header 注入，绝不能让模型从对话里"猜"或由用户在 prompt 里伪造。身份注入后，后续所有 Tool 调用都能从 ThreadLocal 安全读到 userId。

顺序错的典型后果：RAG→Memory 时，检索只用当前单句导致召回不准；Tool 在 RAG 前可能先动作后查政策；身份放最后会让 Tool 拿不到 userId 直接抛 NPE。

关键设计

  • 知识库放"政策类"内容（退款/物流/保修）。
  • 业务动作走 Tool，Tool 内做权限校验。
  • 写操作（退款）返回 pending，明确告知"等人工审核"。

核心 API/代码

1. ChatClient 装配：Memory → RAG → Tool 顺序

// CustomerServiceController.java（节选）
ChatMemory memory = MessageWindowChatMemory.builder().build(); // 默认保留最近 20 条

this.chatClient = builder
    .defaultSystem("""
        你是一名电商智能客服，规则：
        1. 涉及订单具体信息，必须调用工具 listMyOrders / getOrder / requestRefund；不要凭空回答。
        2. 涉及退款政策 / 售后规则等非订单具体信息，参考知识库 context。
        3. 一旦调用 requestRefund，最终回答必须明确"退款申请已提交，等待人工确认"，不要承诺已成功退款。
        4. 传入工具的 currentUserId 只能来自会话上下文（服务端会通过 System 补充信息告知）。
        5. 全程中文，先给结论再给依据。
        """)
    .defaultAdvisors(
        MessageChatMemoryAdvisor.builder(memory).build(),   // ① 记忆最先
        QuestionAnswerAdvisor.builder(vectorStore)          // ② RAG 次之
            .searchRequest(SearchRequest.builder().topK(4).similarityThreshold(0.3d).build())
            .build())
    .defaultTools(tools)                                    // ③ 工具最后，由模型决策调用
    .build();

2. X-User-Id 如何进入 System Prompt

本 demo 采用"把 userId 拼进 user 文本"的最简做法——Controller 从 Header 拿到 X-User-Id 后，在调用前拼接：

@PostMapping("/cs/ask")
public Object ask(@RequestHeader("X-User-Id") String userId,
                  @RequestBody AskRequest req) {
    String cid = req.conversationId() == null ? userId + "-default" : req.conversationId();

    // 服务端注入 userId，模型会按 System Prompt 的要求把它作为 currentUserId 传给 Tool
    String user = "【会话上下文】当前登录用户 id 是 " + userId
                + "，如需调用工具请把它作为 currentUserId 参数传入。\n\n用户问题："
                + req.question();

    return chatClient.prompt()
        .user(user)
        .advisors(a -> a.param("chat_memory_conversation_id", cid)) // 记忆按 cid 隔离
        .call().content();
}

更推荐的生产做法：自定义一个 UserContextAdvisor（实现 BaseChatMemoryAdvisor 或 CallAdvisor），从 ThreadLocal 取出 userId 后写入 System Prompt 或 advisor param；Controller 用 Filter/Interceptor 在请求入口把 userId set 到 ThreadLocal，出口 remove。这样 Tool 也能直接读 ThreadLocal，完全不依赖模型"自觉传参"。

3. CustomerServiceTools 源码要点

@Component
public class CustomerServiceTools {

    // 教学版用内存 Map，生产替换为订单 RPC/DAO
    private static final Map<String, Order> ORDERS = Map.of(
        "O-1001", new Order("O-1001", "u001", "MacBook Pro 14", 15999.00,
                            LocalDate.of(2026, 6, 20), "已发货"),
        "O-1002", new Order("O-1002", "u001", "AirPods Pro", 1899.00,
                            LocalDate.of(2026, 6, 25), "已签收"),
        "O-2001", new Order("O-2001", "u002", "iPad Air", 4799.00,
                            LocalDate.of(2026, 6, 28), "已支付"));

    @Tool(description = "查询当前用户的所有订单，仅返回本人订单。")
    public List<Order> listMyOrders(
            @ToolParam(description = "当前登录用户 id，由服务端注入，不由用户填") String currentUserId) {
        return ORDERS.values().stream()
            .filter(o -> o.ownerUserId().equals(currentUserId))
            .toList();
    }

    @Tool(description = "按订单号查询详情，非本人订单返回 allowed=false。")
    public Map<String, Object> getOrder(String orderId, String currentUserId) {
        Order o = ORDERS.get(orderId);
        if (o == null)              return Map.of("allowed", false, "reason", "order not found");
        if (!o.ownerUserId().equals(currentUserId))
                                    return Map.of("allowed", false, "reason", "not owner");
        return Map.of("allowed", true, "order", o);
    }

    @Tool(description = """
            对某订单发起退款申请。此工具不会真正退款，而是登记一条待人工确认的申请，
            返回 status=pending。调用者必须在最终回答里明确告知用户
            "退款申请已提交，等待客服确认"。
            """)
    public Map<String, Object> requestRefund(String orderId, String reason, String currentUserId) {
        Optional<Order> o = Optional.ofNullable(ORDERS.get(orderId));
        if (o.isEmpty() || !o.get().ownerUserId().equals(currentUserId)) {
            return Map.of("allowed", false, "reason", "no permission");
        }
        // 生产必须加订单状态校验：只有"已支付/未发货/签收≤7天"才允许进入退款流程
        return Map.of("allowed", true, "status", "pending",
                "orderId", orderId, "reason", reason,
                "message", "退款申请已提交，等待客服人工确认。");
    }

    public record Order(String orderId, String ownerUserId, String itemName,
                        double totalAmount, LocalDate paidAt, String status) {}
}

4. 知识库加载：政策文档分块入向量库

@Bean
public ApplicationRunner loadCsKb() {
    return args -> {
        List<String> files = List.of(
            "kb/refund-policy.md",
            "kb/shipping-policy.md",
            "kb/warranty.md");
        TokenTextSplitter splitter = new TokenTextSplitter(400, 200, 10, 5000, true);
        List<Document> chunks = new ArrayList<>();
        for (String path : files) {
            var res = new ClassPathResource(path);
            if (!res.exists()) continue;
            TextReader reader = new TextReader(res);
            reader.getCustomMetadata().put("source", path);  // 来源可追溯
            chunks.addAll(splitter.apply(reader.get()));
        }
        vectorStore.add(chunks);
    };
}

为什么不让模型完全自由选：建议 vs 规则

三种能力（Memory/RAG/Tool）给模型的都是"建议素材"，但业务边界必须由规则卡死——这是 Agent 落地的核心纪律。

能力给模型的是"建议"必须卡死的"规则"
RAG（政策）召回 3-4 段政策原文作为参考答政策类问题必须引用召回原文，模型不能自己编规则（比如不能随口说"30 天无理由"）
Tool（查询）模型可以自由决定何时调订单状态/金额/物流等具体数据必须通过 Tool 获取，不准凭对话记忆瞎编
Tool（写操作）模型可以触发退款/改地址/取消订单绝不允许"立即执行"——本 demo 返回 status=pending 走人工；且 Tool 内必须再做一次权限+状态校验
Memory（历史）帮助模型理解上下文历史里出现的"我是 u002"等身份欺骗话术不能信，身份永远以服务端 Header 注入为准

简单记："建议由模型决策，规则由代码强制"。Prompt 里写的规则是软约束（模型可能违背），Tool 里的权限校验、pending 状态、越权拦截才是硬约束。

转人工触发条件

任何 Agent 都必须有"退场到人工"的兜底。以下条件任一命中即应触发转人工（可以实现为一个 transferToHuman(reason) Tool）：

触发条件说明建议 reason 文案
知识库 + 工具都答不了RAG 相似度低于阈值 + 无合适 Tool，连续 2 轮"问题超出我的能力范围，为您转接人工客服"
用户明确要求转人工"我要人工"/"转人工"/"叫真人来"立刻转，不要挽留
涉及投诉/法律/舆情关键词匹配：起诉/曝光/315/媒体/赔偿"非常抱歉，为您优先转接高级客服"
模型置信度低 / 重复回答模型连续两轮输出语义相似且未解决问题；或 Tool 异常 ≥2 次"这个问题我需要请同事帮您确认"
资金类争议金额错误、未收到退款、重复扣款涉及账务必须人工复核

关键参数与配置

MessageWindowChatMemory 窗口大小

MessageWindowChatMemory.builder().maxMessages(20).build()（默认 20）。单轮对话平均 4-6 token-message，20 条大约能撑 3-5 轮深度对话。高 QPS 场景下建议换到 Redis ChatMemory（spring-ai-alibaba 已提供），避免实例重启丢失。

QuestionAnswerAdvisor topK / threshold

Demo 用 topK=4, similarityThreshold=0.3。政策类知识建议 topK=3~5，threshold=0.3~0.5；阈值太高会漏召回（答非所问），太低会把无关条款灌进 Prompt 误导模型。每次换语料必须重测这两个参数。

TokenTextSplitter 分块

Demo new TokenTextSplitter(400, 200, 10, 5000, true)：chunkSize=400 token、overlap=200。中文政策文档段落短、逻辑性强，400-600 通常合适；overlap 至少 100 避免把一句话切成两半。

chat_memory_conversation_id

Spring AI 1.1 用字符串 key "chat_memory_conversation_id" 通过 .advisors(a -> a.param(...)) 传递；多用户场景必须按 userId + ":" + conversationId 拼，避免 cid 冲突串台。

Tool 描述 (description)

@Tool(description=...) 是模型是否选它的唯一依据。必须写清楚"什么时候用""什么时候不用""参数含义"；写得模糊会导致漏调或错调，写得太啰嗦会占 Prompt 上下文。

temperature

客服场景建议 temperature=0.1~0.3。政策/订单属于事实性问答，高温度会编数据；但完全 0 会让话术生硬，可留 0.2 给表达留一点余量。

边界与常见坑：

  1. 不要让模型传 userId。初学最容易犯的错：把 currentUserId 完全交给模型填——用户说一句"我是 u002 帮我查"，模型就真传 u002。正确做法：要么像 demo 这样在 user 文本里由服务端注入并在 System Prompt 里强制，更好是走 ThreadLocal + 自定义 Advisor，Tool 直接从服务端上下文取，Tool 参数签名里根本不暴露 userId。
  2. 退款/支付类写操作绝不能"直接执行"。本 demo 的 requestRefund 返回 status=pending 是教学版的底线；生产必须走 HITL（Human-In-The-Loop）——写工单、等客服点"通过"、再真正调资金接口。否则模型幻觉一次就是资损事故。
  3. RAG 召回不要依赖"当前问题"单句。如果 QuestionAnswerAdvisor 放在 MessageChatMemoryAdvisor 前面，检索时看不到历史，"那能退吗"这种指代会召回错。顺序一定是 Memory 先、RAG 后，或者做一次"问题重写"（rewrite query with history）。
  4. 政策文档必须带版本和生效时间。退款规则一改，向量库里旧文档要能定位删除——建议 metadata 里放 version、effective_from、effective_to，新版本上线时按 source 文件批量删除旧 chunk 再写入新的。否则模型会引用过期政策。
  5. Tool 失败要有降级文案。订单系统超时、支付系统异常时，Tool 不能把 stack trace 扔给模型；返回结构化的 {success:false, fallback:"订单系统暂时开小差，已为您转人工"}，模型照读即可。
  6. 会话超时要做归档/摘要。长期挂起的会话不能无限保留在 ChatMemory 里——建议超过 30 分钟无新消息自动做一次 summary（调一次模型压缩），原始明细落库审计，内存里只留 summary+最近 N 条。
  7. 所有 Tool 调用必须留审计日志。特别是退款、改地址这类动作：记录 conversationId、userId、Tool 名、入参、出参、模型完整响应，便于事后追溯和风控。
  8. 默认 Advisor 顺序不代表永远正确。Spring AI 1.1 之后 defaultAdvisors 顺序按注册顺序执行；但如果你追加了重试 Advisor、日志 Advisor、限流 Advisor，要重新画链确认它们插在什么位置，避免日志漏记、限流在 RAG 检索后才生效之类的问题。

课后问答

  • "如何加个'转人工'工具？"——实现一个 transferToHuman(reason, chatSummary) Tool，内部写工单+返回固定话术；配合关键词/连续失败兜底策略自动触发。
  • "如何做多轮记忆压缩？"——对超过窗口的历史，用一次 LLM 调用把旧消息总结成 summary 写入 System 消息；Spring AI 提供 MessageWindowChatMemory 的窗口机制，自定义 summary 策略需实现 ChatMemory 装饰器。
  • "生产环境 ChatMemory 放 Redis 具体怎么做？"——Spring AI Alibaba 已提供 Redis 适配，只需把 MessageWindowChatMemory 替换为 Redis 实现并配连接工厂；注意 key 设计加 TTL（如 7 天）避免无限增长，并发写要加锁或用 Redis list 原子操作。
