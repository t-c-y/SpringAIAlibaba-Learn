Lesson 0001 · Spring AI Alibaba

从环境检查到最小 ChatBot

本课的目标很小：把你已经拥有的 Spring Boot 后端经验，连接到 Spring AI Alibaba 的第一个业务接口。

配套 Demo：demos/01-environment-check、demos/02-chatbot。参考卡片：ChatClient 快速参考。

你今天要获得的一个能力

能力目标：看到一段 Spring AI Alibaba 的 ChatClient 代码时，能说清楚它如何把 HTTP 请求转成一次大模型调用。

你已经熟悉 Java / Spring Boot，所以我们不从 Controller、Maven、REST API 重新讲起。你只需要把下面这条新链路装进已有后端知识里：

HTTP 请求
  ↓
Spring Controller
  ↓
ChatClient
  ↓
Spring AI ChatModel 抽象
  ↓
Spring AI Alibaba DashScope 实现
  ↓
DashScope / 百炼模型服务

为什么先做环境检查，再做 ChatBot？

AI 应用的第一类问题通常不是“业务逻辑写错”，而是调用链路没有打通：API Key 没配、模型名写错、账号没权限、网络不通、依赖版本不匹配。

阶段一：环境检查

/env/status 不调用模型，只检查本地配置；/env/ping 真正调用一次模型。

阶段二：最小 ChatBot

/chat/ask 开始接收用户问题，并返回模型回答。它才是业务接口的雏形。

企业开发里，这个顺序很重要：先证明基础设施可用，再叠加业务能力。

自动配置魔法：ChatClient.Builder 哪来的

在 Demo 的 Controller 里你看到的是构造器注入一个 ChatClient.Builder，但整个项目没有任何 @Bean 方法去 new 它。这不是黑魔法，是 Spring Boot Starter 的标准自动装配。

你的 pom.xml 引入 spring-ai-alibaba-starter-dashscope
        │
        ▼
Spring Boot 启动时扫描 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
        │
        ▼
加载 DashScopeAutoConfiguration（Spring AI Alibaba 提供）
        │
        ├─ 根据 spring.ai.dashscope.api-key 创建 DashScopeChatModel（ChatModel 实现）
        ├─ 根据 spring.ai.dashscope.chat.options 创建默认 ChatOptions（model、temperature 等）
        └─ 把 ChatModel 包装成一个 ChatClient.Builder Bean 放进容器
        │
        ▼
你的 Controller 构造器参数 ChatClient.Builder 被自动注入

为什么不自己 new ChatClient(...)？因为 ChatClient.Builder 已经持有了全局的 ChatModel、默认 ChatOptions、ObservationRegistry（可观测性）、MessageConverter 等基础设施。自己 new 等于绕开 Starter 的全部默认装配，一旦要切模型、加观测、改超时，就得自己维护这些 Bean——企业项目里这几乎总是错的。

换句话说，spring-ai-alibaba-starter-dashscope 干的事和 spring-boot-starter-jdbc 干的事是同一类：用约定替代样板代码，把厂商 SDK 的初始化、错误映射、配置绑定都收敛到自动配置里。你只管注入、调用。

核心代码只看四步

String answer = chatClient.prompt()
        .user(request.question())
        .call()
        .content();
  1. prompt()：开始构造一次模型请求。
  2. user(...)：放入用户本次问题。
  3. call()：执行同步模型调用。
  4. content()：取出模型回答文本。

注意：这里还没有 Memory，所以它不是多轮对话；还没有 Tool Calling，所以模型不能真正操作外部系统；还没有 RAG，所以模型不会自动查你的企业知识库。

System Prompt 是默认策略

在 demos/02-chatbot 中，构造方法里设置了 defaultSystem(...)。它不是用户问题，而是这个 ChatBot 的默认角色和行为边界。

this.chatClient = chatClientBuilder
        .defaultSystem("你是一个企业级 Java 后端开发学习助手...")
        .build();

你可以把它类比为后端服务的“默认策略配置”：用户每次问什么会变，但系统希望助手遵守的身份、风格、边界相对稳定。

异常与错误返回：三种典型失败，为什么要先调 /env/ping

大模型调用失败的表现和你熟悉的数据库连接失败很不一样——它不是启动时就报错，而是第一次调用时才抛异常。下面是企业项目里最常见的三种失败模式：

失败原因典型异常 / 返回何时暴露排查手段
API Key 错误 / 未配置 401 Unauthorized 或 DashScope SDK 抛认证异常，消息里含 "Invalid API key" / "Unathorized" 第一次真正调用模型时（/env/ping 或 /chat/ask） 先调 /env/status 确认 apiKeyConfigured=true，再调 /env/ping
模型名写错 / 账号无权访问 400 Bad Request 或 404 ModelNotExist，消息含 "model not exist" / "permission denied" 第一次调用模型时 去百炼控制台核对模型 ID（如 qwen-plus、qwen-turbo），确认已开通
网络超时 / 限流 ReadTimeoutException 或 HTTP 429 Too Many Requests，5xx 业务高峰期或弱网环境下偶发 配置超时时间与重试；查看 DashScope 控制台配额与 QPS

这就是为什么 Demo 里要先提供 /env/ping——/env/status 只读本地配置，不发 HTTP，能证明「Key 变量存在」但不能证明「Key 有效、网络通、模型名正确」。/env/ping 才是真正的烟雾测试（smoke test），它以最小代价（一次 1-token 调用）打通整条链路。企业项目在 K8s 就绪探针、CI 部署后校验都应该复用这个模式。

核心 API 完整示例

下面是 Demo 02 中 ChatController 的关键骨架，包含构造器注入、默认 System Prompt、同步调用、请求记录、响应封装，可直接作为你自己项目的起点：

@RestController
@RequestMapping("/chat")
public class ChatController {

    private final ChatClient chatClient;

    // ChatClient.Builder 由 Starter 自动装配注入，无需 @Bean
    public ChatController(ChatClient.Builder chatClientBuilder) {
        this.chatClient = chatClientBuilder
                .defaultSystem("你是一个企业级 Java 后端开发学习助手，"
                        + "回答要简洁、代码优先、不编造不存在的 API。")
                .build();
    }

    @GetMapping("/status")
    public Map<String, Object> status() {
        // 读本地配置即可，不调用模型
        return Map.of(
                "model", chatClient.getChatOptions().getModel(),
                "apiKeyConfigured", StringUtils.hasText(
                        DashScopeApiKeyUtils.getConfiguredApiKey())
        );
    }

    @PostMapping("/ask")
    public ChatResponse ask(@RequestBody ChatRequest request) {
        String answer = chatClient.prompt()
                .user(request.question())    // 用户问题
                .call()                      // 同步调用，阻塞等待
                .content();                  // 取文本回答（抛异常时此处会冒泡）
        return new ChatResponse(true, "qwen3.7-max",
                request.question(), answer);
    }

    // 生产环境建议在此加 @ExceptionHandler，把 DashScope 异常
    // 转成稳定的业务错误码返回给前端，而不是暴露 500 原始栈
}

关键配置项

Demo 里用了 ${DASHSCOPE_CHAT_MODEL:qwen3.7-max} 这种写法。它是 Spring 占位符语法：先读环境变量 / 系统属性 DASHSCOPE_CHAT_MODEL，读不到就用冒号后的默认值 qwen3.7-max。企业里通常用环境变量（容器、K8s ConfigMap/Secret、CI）覆盖默认值，本地开发则用默认值直接跑。

配置项（application.yml）环境变量占位符写法默认值什么时候需要调
spring.ai.dashscope.api-key ${DASHSCOPE_API_KEY} 无（必填） 必填。生产走 Secret / 配置中心，不要写进 yml
spring.ai.dashscope.chat.options.model ${DASHSCOPE_CHAT_MODEL:qwen3.7-max} qwen3.7-max 要切更强模型（qwen-max）、更快更便宜模型（qwen-turbo / qwen-plus）或多模态时改
spring.ai.dashscope.chat.options.temperature 通常写死，不注入环境变量 0.7（以 Starter 默认为准） 0 适合事实回答/代码生成；接近 1 适合创意写作。企业场景常见 0~0.3
spring.ai.dashscope.base-url ${DASHSCOPE_BASE_URL:https://dashscope.aliyuncs.com/api/v1} 官方地址 走代理、VPC 专线或私有化部署时改
连接/读超时 Spring WebClient / RestClient 超时配置（示意） Spring 默认 大模型 P99 高，建议读超时 ≥ 30s；流式可更长

提示：所有 spring.ai.dashscope.* 的配置都由 DashScopeAutoConfiguration 在启动时绑定。你可以在 application.yml 里写默认值，但生产环境敏感项（api-key）必须通过 Secret / 环境变量注入，不要提交进 Git。

边界与常见坑

坑 1：API Key 硬编码进 yml 并提交到仓库。最常见也最致命。团队越大、仓库越公开，泄露面越大。统一用环境变量 + .gitignore 掉本地 application-local.yml，生产用云厂商 Secret / KMS / 配置中心加密字段。

坑 2：把 call() 放在事务里或接口主线程里不设超时。大模型 P99 延迟可达数秒到十几秒，而 Spring 事务默认超时通常较短；不加超时会导致 HTTP 线程池被打满、事务长占连接。建议：同步接口设置读超时；高并发场景用 stream() 或异步。

坑 3：以为 /env/status 显示 apiKeyConfigured=true 就万事大吉。它只检查了本地变量是否存在，不校验 Key 有效性、余额、模型权限、网络连通性。部署后第一次验证必须调一次 /env/ping（或等价的最小模型调用），否则就是「配置看起来对但模型调不通」的黑盒状态。

坑 4：把默认 System Prompt 当成安全边界。defaultSystem 只是给模型的偏好提示，不是权限系统。用户通过 prompt injection 可能绕过。真正的权限控制必须在 Controller / Service 层用代码判断，不要把安全交给 prompt。
