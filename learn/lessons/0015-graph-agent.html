Graph 客服 Agent：显式状态机 + LLM 节点

本课目标：从"让 LLM 自由发挥"进化到"显式状态机 + LLM 节点"，掌握 Graph 编排思维。

配套 Demo：demos/16-graph-agent。参考卡：Graph 快速参考。

你今天要获得的一个能力：把"让 LLM 自由发挥"的客服 Agent，重构成"显式图状态机 + LLM 节点"。你能得到四个在生产里值钱的东西：

  • 可控：每一次意图识别、每一次跳转都由代码决定，LLM 不再拥有"下一句我想干嘛"的自由。
  • 可审计：trace 数组里的每一行就是一条决策轨迹，出问题能回放、能追责。
  • 可调试：节点是一个普通 Java 方法，可以单测、可以打断点、可以单独替换模型，而不是在一大段 Prompt 里猜哪一句出了错。
  • 可预测：能放 HITL（Human-in-the-Loop）、能加守卫（maxSteps）、能强制走兜底（FALLBACK），P99 行为可预期。

为什么要 Graph

  • 关键动作要求"必须走人工审批"，纯 Prompt 很难保证。
  • 意图判错时要能诊断和回放，需要显式 trace。
  • 把 LLM 用在"擅长的地方"（意图分类、生成话术），把 transition 交给代码。

自由 ReAct Agent vs 图状态机

自由 ReAct Agent

  • 可控性：低，下一步动作由 LLM 按 Prompt 临时决定。
  • 可测试：难，必须跑全链路；单测只能测 Prompt。
  • 调试难度：高，trace 是自然语言 thought，不稳定。
  • 灵活度：高，开放式任务（调研、写代码）效果好。
  • 适用场景：内部工具、个人助理、开放式创作。

图状态机 Agent

  • 可控性：高，边（edge）是代码里的 if/switch。
  • 可测试：易，每个节点是一个方法，可以脱离 LLM 单测。
  • 调试难度：低，trace 是节点名数组，可以直接定位。
  • 灵活度：中，新增能力必须加节点、加边。
  • 适用场景：客服、退款、理赔、工单等有 SOP 的企业流程。

经验法则：业务流有明确步骤或合规要求时选图；开放式"帮我搞定这件事"选 ReAct。两者可以嵌套——图节点里可以调一个 ReAct Agent 处理子任务。

Graph Runtime 核心概念

本课 Demo 用大约 100 行 Java 手写了一个最小 Graph Runtime，把所有核心概念摆到台面上。你理解了这几个概念，以后切 LangGraph / Spring AI 官方 Graph SDK 只是换 API，思路不变。

              ┌───────────┐
              │  CLASSIFY │  ← 唯一用 LLM 的节点（意图分类）
              └─────┬─────┘
        ┌───────┬───┴────┬──────────┬──────────┐
        ▼       ▼        ▼          ▼          ▼
 ┌──────────┐ ┌────────┐ ┌──────────────┐ ┌────────┐ ┌──────────┐
 │ORDER_QUERY│ │POLICY_QA│ │REFUND_CONFIRM│ │SMALL_TALK│ │ FALLBACK │
 └─────┬────┘ └───┬────┘ └──────┬───────┘ └───┬────┘ └────┬─────┘
       └──────────┴──────┬──────┴─────────────┴───────────┘
                         ▼
                       ┌─────┐
                       │ END │  ← 终态，state.finished=true
                       └─────┘

守卫：guard++ < maxSteps（默认 16），超步强制 END 并打告警。
概念是什么Demo 里对应
节点（Node）Function<ChatState, String>，读 state、写 state、返回下一节点名。classify / orderQuery / policyQa / refundConfirm / smallTalk / fallback
边（Edge / Transition）节点的返回值；由代码 switch/if 决定，不走 LLM。classify() 末尾的 return switch(s.intent)。
状态（State Map）共享上下文对象，节点之间唯一的数据通道。ChatState（含 vars: Map<String,Object>）。
根节点 CLASSIFY入口，唯一允许调 LLM 做"判别"的地方。state.node = "CLASSIFY"，classifier ChatClient。
结束节点 END终态标记，state.finished = true，循环退出。if ("END".equals(state.node)) state.finished = true;
maxSteps 守卫防死循环：所有节点总步数 ≤ N，超了直接 END 并打日志。while (!state.finished && guard++ < 16)

运行流程：(1) 初始化 state，node=CLASSIFY；(2) 进入 while 循环，把当前节点名加入 trace；(3) 调 step(state) 分发到对应 handler；(4) handler 读写 state、返回下一节点名；(5) 如果返回 END 或步数超限，退出循环；(6) 按 answerParts 拼接最终回答，连同 trace/intent/vars 一起返回。

关键点：LLM 只做"它擅长的不确定性判断"（意图分类、话术生成），而"下一步去哪"是确定性代码。这就是 Graph 比纯 ReAct 稳定的根本原因。

本课 Graph

CLASSIFY
├─ ORDER_QUERY
├─ POLICY_QA
├─ REFUND_CONFIRM (pending)
├─ SMALL_TALK

关键实现

while (!state.finished && guard++ < 16) {
    trace.add(state.node);
    state.node = step(state);
}

state 显式记录 intent、vars、answerParts，节点之间通过 state 共享数据，最后统一到 END。

CLASSIFY 节点：让分类稳定的 Prompt 技巧

CLASSIFY 是整个图里唯一必须稳定的 LLM 调用，它一旦飘了，后面所有节点全错。Demo 里的 System Prompt 遵循四条军规：

  1. 给封闭枚举：只允许输出 ORDER_QUERY / POLICY_QA / REFUND / SMALL_TALK / UNKNOWN，用代码白名单再校验一次（List.of(...).contains(out) ? out : "UNKNOWN"）。
  2. 显式列出判别规则：每条 intent 对应什么关键词、什么语义，写得清清楚楚，不要让模型"自己体会"。
  3. 禁止多余输出：Prompt 里写明"不要输出多余文字"，并在代码里 .trim()。
  4. temperature = 0：分类是判别任务，要的就是确定性，不要创造性。Spring AI Alibaba 默认可通过 defaultOptions(ChatOptionsBuilder.builder().temperature(0.0).build()) 设置。
.defaultSystem("""
    你是意图分类器。只输出以下枚举之一，不要输出多余文字：
    ORDER_QUERY / POLICY_QA / REFUND / SMALL_TALK / UNKNOWN
    判断规则：
    - 涉及"我的订单/物流/发货"→ ORDER_QUERY
    - 涉及"退款/退货/政策/保修"→ POLICY_QA
    - 明确要求发起退款申请 → REFUND
    - 打招呼/闲聊 → SMALL_TALK
    - 无法判断 → UNKNOWN
    """)

进阶技巧：当意图数 ≥ 6 或者边界模糊时，加 2-3 条 few-shot（用户："我要退钱" → REFUND）准确率会大幅上升；再进一步可以把分类改成"结构化输出"（EntitySpec 或 JSON mode），避免字符串匹配。

State 字段与节点读写约定

State 就是图的"共享内存"。事先约定好每个字段谁读谁写，能避免节点互相踩脚。

字段类型谁写谁读用途
userIdString入口 ask()ORDER_QUERY / REFUND_CONFIRM从 header 透传，做数据查询 key。
userInputString入口 ask()CLASSIFY / POLICY_QA / SMALL_TALK本轮用户原话，避免被后续节点覆盖。
intentStringCLASSIFY入口（仅日志/返回）/ 可被人工纠错覆盖分类结果枚举值。
vars.ordersListORDER_QUERYORDER_QUERY（answerer 生成话术时读）查出的订单列表，结构化缓存。
vars.policyTextStringPOLICY_QA（内嵌）POLICY_QA本 demo 内嵌政策文本，生产里换成 RAG 检索结果。
vars.pendingRefundMapREFUND_CONFIRM下一流程（人工审批系统）挂起的退款申请，等用户/人工确认。
answerPartsList<String>所有业务节点finalAnswer()按加入顺序拼接成最终回复，支持多节点分段输出。

纪律：(1) 字段要声明"谁写"，多个节点写字段要有先后顺序；(2) vars 是兜底的 Map，但优先用强类型字段，IDE 能帮你发现错误；(3) 长会话要做 state 摘要，别把所有历史都塞进去。

节点设计粒度原则

一个节点做一件事

节点是"动作"不是"功能模块"。classify 只做分类，orderQuery 只查订单+总结。不要让一个节点里出现"先分类、再查库、再决定要不要退款"这种代码。

I/O 走结构化数据

节点之间通过 ChatState 的字段或 vars 传对象（List/Map/Record），不要传自然语言。classify 输出是 enum 字符串，不是"我觉得用户想问订单"。

节点可单测

每个节点是一个 private String xxx(ChatState s) 方法。单测时可以 new ChatState、手动塞字段、直接调节点方法、断言 state.vars 和返回的下一个节点名——不需要启动 LLM。

业务节点不调 LLM 判流程

ORDER_QUERY / POLICY_QA 里 LLM 只做"话术生成"，流程跳转（返回 END 还是别的节点）由代码决定。只有 CLASSIFY 用 LLM 做判别，降低不确定性的爆炸半径。

REFUND_CONFIRM 与 HITL（Human-in-the-Loop）

退款、打款、发货这类资金/履约动作，不能让 LLM 一句话就执行。本课 Demo 做了一个教学版 HITL：

private String refundConfirm(ChatState s) {
    s.vars.put("pendingRefund", Map.of(
        "requestBy", s.userId, "rawInput", s.userInput));
    s.append("已识别到退款诉求，退款申请已进入人工审批队列，"
           + "客服将在 24 小时内联系您确认。此过程不会自动打款。");
    return "END";
}

这里其实是"半 HITL"——图直接跑完一轮，把挂起信息塞进 pendingRefund，等人工系统异步处理。真正的 HITL 需要：

  1. 挂起（suspend）：图跑到 REFUND_CONFIRM 就停，把 state 持久化（DB/Redis），返回给前端"请确认是否退款 [是/否]"。
  2. 等待用户输入：前端把用户的"确认/取消"作为新一轮请求，带上图实例 ID，恢复 state。
  3. 恢复（resume）：从持久化里读回 state，根据用户输入走 REFUND_EXECUTE 或 REFUND_CANCEL 分支，继续跑到 END。

LangGraph / Spring AI 官方 Graph SDK 都会提供 interrupt() 这类原生挂起/恢复 API。本课手写版没实现，但你要知道：挂起点就是"节点返回一个特殊值（不是 END 也不是下个节点名，而是 WAIT_INPUT）"，Runtime 据此持久化 state。

与 LangGraph / Spring AI 官方 Graph SDK 的关系

Spring AI Alibaba 1.0.0.2 时代，社区里 Java 生态的 Graph 编排 SDK 还在快速演进；Python 侧 LangGraph 已经成熟。你现在手写这个最小 Runtime，有三个好处：

  • 零额外依赖：不用等 SDK 稳定，今天就能在生产里跑。
  • 完全可控：trace、超时、挂起、人工纠错都可以按自己业务定制，不会被框架抽象层卡住。
  • 易调试：step() 方法就是一个 switch，打个断点什么都看得清。

未来等 Spring AI 官方 Graph（或社区封装的 LangGraph4j 之类）稳定后迁移，结构不变——节点还是"读 state、写 state、返回下一节点"，只是把 step() 的 switch 换成框架的 .addNode("classify", ...).addEdge(...) 声明式 API。你今天投入的设计资产（节点拆分、state 字段、HITL 边界）全部可复用。

关键参数与配置

参数默认值（Demo）何时调
maxSteps（guard 上限）16节点数 × 2，留足重试/兜底空间；资金类场景调到 节点数 + 2，走超就告警，不要给 LLM 太多"自救"机会。
classifier 的 temperature默认（建议显式设 0）分类/判别类永远 0；只有生成话术的 answerer 可以给 0.3-0.7。
classifier 模型与 answerer 相同生产里换成更便宜、更快的小模型（如 qwen-turbo），准确率够就行，把 qwen-max 留给话术生成。
policyText 来源硬编码字符串超过 10 条政策、或需要运营同学改，就接 RAG（VectorStore）；不要继续堆硬编码。
orders 数据源内存 Map（Mock）换成真实订单 RPC/DB 查询节点；加超时、熔断、fallback 数据。
state 存储请求内内存要做 HITL / 多轮挂起时，state 必须持久化（Redis/DB），key 用 conversationId。

边界与常见坑

  1. 节点不要无限膨胀。超过 15 个节点图就变成"代码迷宫"，新人看不懂、改不动。节点数过多时，把子图抽成独立流程（例：把"退款-退货-换货"抽成 after-sale subgraph）。
  2. 必须有 FALLBACK 兜底节点。任何分类失败、工具调用异常、模型返回非枚举，都要能走到 FALLBACK 返回"请联系人工"，绝对不能把 LLM 的 raw 输出直接给用户。
  3. maxSteps 必须有限，并打超步日志。生产里 guard == maxSteps 要打 ERROR 日志（带 trace 和 state 快照），因为这意味着"图在两个节点间死循环"或"LLM 一直返回 UNKNOWN"，是 bug 信号。
  4. 分类错误要有"人工纠错"通道。classifier 准确率不可能 100%。要提供运营/客服后台，能把某条 trace 的 intent 从 ORDER_QUERY 改成 POLICY_QA，反哺 few-shot 或微调数据。
  5. state 不能无限膨胀。多轮对话里 vars 里塞几十轮历史会让 token 爆炸、OOM。长会话要做"对话摘要"节点，定期把旧轮次压成一段 summary。
  6. 每一次流转都打 trace。本 Demo 返回的 trace 字段不是摆设——生产里要把它落到日志/链路追踪里（含 userId、intent、耗时、模型输入输出长度），出问题才能回放。
  7. 资金/退款/发货节点必须 HITL。REFUND_CONFIRM 里不能直接调退款 RPC。凡涉及真金白银或不可逆动作，一定要有显式的"确认"步骤和人工兜底，哪怕只是"弹个二次确认框"。

课后问答

  • "什么时候用 Graph，什么时候用 Advisor + Tool（ReAct）？我的客服场景能只写一个 ChatClient 吗？"——回到上面的对比表，只要有"必须人工审批""必须按 SOP 走""出问题要追责"三条中的任何一条，就该用 Graph。
  • "Graph 里怎么插入人工确认？怎么让图跑到一半停下来，等用户点'确认退款'再继续？"——参考 REFUND_CONFIRM 一节：生产里要做 state 持久化 + interrupt/resume 机制，节点返回特殊"WAIT_INPUT"信号，下一轮带 conversationId 恢复。
  • "我的客服有 30 个意图，CLASSIFY 一次分不准怎么办？"——把一级分类粗化（订单/售后/账户/其他），进子图再用二级分类；或者改成"层级分类器"（先粗分再细分），每一级都用小模型 + 白名单校验，错误会比"一次 30 分类"低很多。
