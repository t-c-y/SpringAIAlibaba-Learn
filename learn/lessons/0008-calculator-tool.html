Tool Calling 入门：计算器

本课目标：第一次让模型"调用你写的 Java 方法"，理解 @Tool 描述如何影响模型选择。

配套 Demo：demos/09-calculator-tool。参考卡：Tool Calling 快速参考。

你今天要获得的一个能力：能独立写出被大模型稳定选中并正确调用的 @Tool 方法——理解"一次对话"背后隐藏的多轮 HTTP 往返，能通过调 description 让工具命中率从"偶尔用"提升到"必中"，并能按生产规范处理幂等、参数校验、错误返回和审计。

Tool Calling 内部机制：一次对话背后的多轮往返

用户看到的是"我问了一句，模型答了一句"，但开启 Tool Calling 后，Spring AI 与 LLM 之间可能发生多次 HTTP 往返，直到模型给出最终自然语言答案。下面是一次四则运算 (3.14+2.86)*5 的真实时序：

用户 " (3.14 + 2.86) * 5 等于多少 "
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│ ① ChatClient 把 user message + tools(JSON Schema)       │
│    POST 给 DashScope（第 1 次 HTTP 往返）                │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│ ② LLM 判断：括号里是加法 → 不需要最终答案                │
│    返回 finish_reason=tool_calls                         │
│    tool_calls: [{id=t1, name=add, args={a:3.14,b:2.86}}]│
└─────────────────────────────────────────────────────────┘
  │  Spring AI 反射调用 CalculatorTools.add(3.14, 2.86)
  ▼  拿到 6.0，封装成 role=tool 的消息
┌─────────────────────────────────────────────────────────┐
│ ③ 把 user+assistant(tool_calls)+tool(6.0) 一起 POST     │
│    （第 2 次 HTTP 往返）                                 │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│ ④ LLM 判断：还要乘 5 → finish_reason=tool_calls          │
│    tool_calls: [{id=t2, name=multiply, args={a:6,b:5}}] │
└─────────────────────────────────────────────────────────┘
  │  Spring AI 反射调用 multiply(6,5) → 30.0
  ▼
┌─────────────────────────────────────────────────────────┐
│ ⑤ 再次 POST（第 3 次 HTTP 往返）                         │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│ ⑥ LLM 判断：结果已齐 → finish_reason=stop                │
│    content: "(3.14+2.86)*5 = 30"                        │
└─────────────────────────────────────────────────────────┘
  │
  ▼
用户看到一次完整回答，背后其实是 3 次 HTTP 调用、2 次本地 Java 反射。

关键认知：Tool Calling 不是"模型直接调你的 Java"，而是模型用结构化 JSON 告诉框架"我想用 xxx 方法、参数是 yyy"，框架负责反射执行、把结果以 role=tool 的消息塞回消息历史，再让模型继续推理。这是一个 "LLM 决策 → 框架执行 → 结果回传 → LLM 再决策" 的循环，直到 finish_reason=stop。

这也是为什么"问一句话"会在日志里看到多次 tool call、多次 HTTP 请求——越复杂的表达式、越多步的推理，往返次数越多。延迟不是一次 RTT，而是 N 次 RTT + N 次本地方法执行之和。生产环境评估 SLA 时必须按"最坏往返次数"估延迟。

Spring AI 的 ChatClient 默认开启工具循环（内部由 ToolCallingChatClientAdvisor 驱动），你不需要手写 while 循环；但你要知道这个循环存在，才能理解为什么工具方法可能被重试、被重复调用，以及为什么抛异常会打断整个循环。

信息流

用户 → ChatClient → LLM
                    │  选工具、传参数
                    ▼
                  你的 Java 方法（@Tool）
                    │  返回结果
                    ▼
                  LLM 综合成自然语言

核心注解与完整源码

Spring AI 通过两个注解把普通 Java 方法暴露给模型：@Tool 标在方法上，@ToolParam 标在参数上。框架在启动时扫描它们，生成 OpenAI/DashScope 兼容的 JSON Schema 放进 tools 字段发给模型。模型从未见过你的源码，它选ԅ只能靠 description 的文字做"工具选择"——description 写得差 = 工具选不中 / 参数传错，这是企业里 80% Tool Calling 问题的根因。

本 Demo 的 CalculatorTools.java（四个原子方法，每个做一件事）：

package com.example.springaialibaba.calculatortool;

import org.springframework.ai.tool.annotation.Tool;
import org.springframework.ai.tool.annotation.ToolParam;
import org.springframework.stereotype.Component;

@Component
public class CalculatorTools {

    @Tool(description = "两个数相加，输入 a 和 b，返回 a + b。")
    public double add(@ToolParam(description = "第一个加数") double a,
                      @ToolParam(description = "第二个加数") double b) {
        return a + b;
    }

    @Tool(description = "两个数相减，返回 a - b。")
    public double subtract(@ToolParam(description = "被减数") double a,
                           @ToolParam(description = "减数") double b) {
        return a - b;
    }

    @Tool(description = "两个数相乘，返回 a * b。")
    public double multiply(@ToolParam(description = "第一个乘数") double a,
                           @ToolParam(description = "第二个乘数") double b) {
        return a * b;
    }

    @Tool(description = "两个数相除，返回 a / b。除数为 0 时抛异常并向调用方返回原因。")
    public double divide(@ToolParam(description = "被除数") double a,
                         @ToolParam(description = "除数，不能为 0") double b) {
        if (b == 0) {
            throw new IllegalArgumentException("除数不能为 0");
        }
        return a / b;
    }
}

注册到 ChatClient（Controller 里通常这么写）：

@Bean
public ChatClient calculatorClient(ChatClient.Builder builder,
                                   CalculatorTools tools) {
    return builder
            .defaultTools(tools)           // 扫描 @Tool 方法注册
            .defaultAdvisors(new MessageChatMemoryAdvisor(/*...*/))
            .build();
}

description 的写法原则

  • 写"何时用"+"做什么"+"返回什么"，不要只写方法名翻译。例："查询某员工在指定月份的工资明细，返回基本工资+奖金+扣款" 比 "工资查询" 命中率高一个数量级。
  • 用中文、别中英混杂、别用缩写。模型对 "orderId" 可能理解，但 "order_no" "ord_id" "OID" 混合出现会显著降低命中率。
  • 参数 description 也要写——它会进 JSON Schema 的 description 字段，模型靠它决定自然语言中的哪片字对应哪个参数。
  • 一个方法只做一件事。别写 compute(String expr) 试图让模型传表达式字符串——模型不知道你表达式语法，参数错误率极高。

@ToolParam 类型反序列化：自然语言怎么变成 double

当用户说"三点一四加二点八六"，模型要在 tool_calls 里输出 {"a": 3.14, "b": 2.86}，Spring AI 再用 Jackson 反序列化到方法参数。Spring AI 对 @ToolParam 支持的类型：

Java 类型模型传参示例注意事项
int / Integer5传小数会反序列化失败
long / Long1700000000000注意 JS number 精度（超过 2^53 要 String）
double / Double3.14自然语言"三点一四"模型会转 3.14
boolean / Booleantrue / false模型会把"是/否/对/错"映射过来
String"张三"首选，最稳
List<String> / List<T>["北京","上海"]适合多选场景
Map<String,Object>{"k":"v"}兜底，少用，模型容易乱填 key
嵌套 record / POJO{"addr":{"city":"bj"}}会递归生成 JSON Schema
enum"PAID"枚举值会进 schema enum 字段，强烈推荐

建议：能用 String + 业务校验就不用复杂对象；能用 enum 就不用自由字符串；金额/长 ID 一律用 String 避免精度损失。

关键参数与配置

spring.ai.dashscope.api-key

默认：无
何时调：必填。建议走环境变量 DASHSCOPE_API_KEY，别写进 yml。

spring.ai.dashscope.chat.options.model

默认：qwen-plus
何时调：Tool Calling 推荐 qwen-max/qwen-plus；qwen-turbo 在多步工具上稳定性差。

temperature

默认：0.7
何时调：Tool 场景一律降到 0.1~0.3，降低乱选工具、乱造参数的概率。

defaultTools(tools...)

默认：空
何时调：注册 @Tool 对象。生产环境不要无脑 defaultTools 全部工具，按接口最小集注册。

toolCallBack / 工具返回值类型

默认：toString 序列化
何时调：建议返回 record 或明确结构的 JSON 字符串；抛异常会中断循环。

ChatModel 工具调用的 internalToolCallLimit

默认：10 次
何时调：多步推理超过 10 次会强制 stop；复杂 agent 可适度上调，同时加成本告警。

实验：description 的质量决定工具命中率

做完基础练习后，做这个对比实验（10 分钟，收益巨大）：

  1. 清晰 description 版本（当前代码）：问 1 + 1 = ?，日志里应看到 add 被调用，返回 2.0，命中率接近 100%。
  2. 把 add 的 description 改成 "数学工具"，重启，问同样的问题——模型大概率不再调用 add，而是直接用参数化知识回答，甚至选错方法。
  3. 再改成 "utility method for numbers"（纯英文/缩写），中文模型命中率进一步下降。
  4. 把 add 删掉、保留 subtract/multiply/divide，问 1+1——模型通常会"用 multiply(1,1) 凑合"或直接放弃调用，因为它没有更好的选择。这就是生产里"工具写了但模型不用"的最常见根因：不是模型笨，是你的 description 没给它足够的选择信号。

边界与常见坑（生产必读）：

  1. 工具方法必须无副作用或严格幂等。前面时序图里工具可能被模型在多轮中重复调用，Spring AI 的重试/重连也可能重放 tool message——如果是"扣款""发消息""创建订单"这种副作用方法，重复调用就是事故。生产要么做幂等键（从 tool_call_id 派生），要么把有副作用操作拆到工具外的人工确认流程。
  2. 永远不要信任模型传的参数。模型可能把 "1 OR 1=1" 传进 String 参数、把 -1 传进"数量"字段、把 null 传进必填项。@ToolParam(required=true) 只生成 schema 提示，不做校验；必须在方法内用 Objects.requireNonNull、Bean Validation、业务校验主动拦截，绝不直接拼 SQL/传下游。
  3. 工具错误要返回结构化错误，不要抛异常。把 throw new IllegalArgumentException("除数不能为0") 改成 return new ToolError("DIVIDE_BY_ZERO", "除数 b 不能为 0，请用户重新输入")。抛异常会被 Spring AI 转成一条错误消息塞回模型——有些模型不会"再试一次"，直接把异常堆栈甩给用户；返回结构化错误 record，模型能基于错误码自然语言解释并引导用户。
  4. description 禁止中英混杂、禁止缩写、禁止业务黑话。"查询 OID 对应 ord 的 payAmt"这种描述在中文模型里命中率极低。统一中文、完整词、写清"什么时候用、做什么、返回什么"。
  5. 不要在 Controller 层 defaultTools 把所有工具全暴露。工具是按"对话场景"注册的，不是按 Spring Bean 全局共享。例如"退款工具"只应出现在售后客服 ChatClient 里；如果所有对话都注册了它，任何用户都可能通过诱导让模型调用退款。按接口/场景白名单注册。
  6. 工具调用必须打审计日志。记录 tool_call_id、方法名、参数（脱敏后）、返回值、耗时、用户 ID、会话 ID。没有审计日志，出了"模型乱调工具"事故无从排查。建议用 AOP 环绕 @Tool 方法统一埋点。

课后问答

  • "多个 Tool 描述冲突会怎样？"——模型会在几个工具之间随机选，或者每轮交替用，表现为结果反复波动。解决：合并/差异化 description，不要让两个工具在同一场景都"看起来合适"。
  • "Tool 抛异常时模型看到什么？"——Spring AI 默认把异常 message 包成 tool message 回传，模型通常直接把错误原文说给用户，或者干脆终止回答。生产改为返回结构化错误 record，让模型可以"理解错误并引导用户"。
  • "工具调用次数过多导致延迟高怎么办？"——三个方向：①拆原子工具改为"业务复合工具"（一次 tool call 完成多步）；②在 System Prompt 里明确"能一次性算完就一次调用，不要分步"；③评估换用 tool-calling 更强的模型（qwen-max 比 turbo 少走往返）。
