Lesson 0018 · Spring AI Alibaba

企业级 Agent 平台原型：装配前 17 课的能力

本课目标：把 Agent 抽象化，把多 Agent 路由、限流、审计装配成可扩展骨架。

配套 Demo：demos/19-agent-platform。参考卡：Agent 平台快速参考。

你今天要获得的一个能力：把前 17 课学到的 ChatClient、结构化输出、MCP、RAG、Function Calling、多 Agent 协作等能力，装进一个统一网关——对外只暴露 POST /platform/ask，对内由 RequestGuard（限流/鉴权）+ AgentRouter（按意图路由）+ AuditLog（调用审计）三件套完成路由。学完你能在自己公司里搭出一个最小可扩展的"Agent 中台原型"，理解生产环境 Agent 平台缺什么。

平台网关三件套

从用户 HTTP 请求到拿到答案，统一经过三个横切组件：

RequestGuard · 入口守护者

按 userId 做鉴权 + 限流。教学版使用固定窗口计数（20 req/min/用户），生产替换为 Sentinel 令牌桶/熔断。超量返回 429 Too Many Requests 并带 Retry-After。

AgentRouter · 意图调度

启动时把所有 Agent 实现注入 List<Agent>，用一个便宜的分类模型（如 qwen-turbo）根据用户问题选 Agent id；高频路径加关键词规则前置兜底。

AuditLog · 调用审计

记录 id/at/userId/agentId/latencyMs/question/answer。教学版用 ConcurrentLinkedDeque + 容量上限避免内存泄漏；生产落库/Kafka，后续用于质量分析、badcase 回灌、计费对账。

                ┌────────────────────────────────────────────────┐
   HTTP POST    │  PlatformController.ask(userId, question)      │
  /platform/ask │                                                │
  ───────────►  │  1. RequestGuard.tryAcquire(userId)            │
                │       │  超限 → 429 + Retry-After               │
                │       ▼                                        │
                │  2. AgentRouter.route(question) → Agent        │
                │       │  未命中 → fallback Agent（小模型）      │
                │       ▼                                        │
                │  3. agent.handle(req) → AgentResponse          │
                │       │  超时 → 熔断 + 降级回复                │
                │       ▼                                        │
                │  4. AuditLog.log(record) → 内存/DB/Kafka       │
                │       ▼                                        │
                │  return { success, routedAgent, latencyMs, answer }
                └────────────────────────────────────────────────┘

为什么要三件套拆开？鉴权限流是平台级横切关注点，路由是策略点（新增 Agent 不改入口），审计是合规与改进数据源——任何一个替换都不影响另两个。

架构

HTTP Ingress
  ↓ RequestGuard 鉴权+限流
  ↓ AgentRouter LLM 分类

关键组件原理

1. Agent 接口：统一抽象，所有能力一视同仁

小模型闲聊、FAQ 检索、RAG 问答、客服工单 Agent、代码生成 Agent，对外都只长一个样子。这是平台能"横向扩展"的根本：

public interface Agent {
    String id();                              // 全局唯一 id，路由键
    String description();                     // 供分类模型读的"能力说明"
    AgentResponse handle(AgentRequest req);

    record AgentRequest(String userId, String question) {}
    record AgentResponse(String agentId, String answer, long latencyMs) {}
}

有了统一接口，权限、审计、灰度、超时隔离全部可以挂在入口层做，每个 Agent 作者只关心自己的业务。新 Agent 上线 = 新增一个 @Component 实现类，零改动平台代码（Spring 自动注入 List<Agent>，启动时即注册）。

2. RequestGuard：令牌桶限流（教学版 vs 生产版）

Demo 用的是"每分钟固定窗口计数"（每分钟清 0，20 次上限），简单但有窗口边界突发问题。生产常用令牌桶：

  • bucket capacity = 20（桶容量，即允许瞬时突发）；
  • 每 3 秒补 1 个 token（60 / 20 = 3），匀速填充 = 稳态 20 req/min；
  • 请求来时先拿 token，拿不到直接返回 429 Too Many Requests + Retry-After: N 头；
  • 按 userId / tenantId / 全局 三级粒度分别配桶。
// 语义接口示意（生产用 Sentinel 注解即可）
public record AccessCheck(boolean allowed, String reason, int retryAfterSec) {}

public AccessCheck check(String userId) {
    TokenBucket bucket = buckets.computeIfAbsent(userId,
        k -> TokenBucket.of(capacity=20, refillPerSec=1.0/3));
    if (bucket.tryConsume(1)) return new AccessCheck(true, "ok", 0);
    return new AccessCheck(false, "rate_limited", bucket.secondsToNextToken());
}

Demo 里的固定窗口 20 次/分钟 只是教学：它会在"0 秒 20 次 + 61 秒瞬间又 20 次"的边界上放过 40 次。Sentinel/Resilience4j 的 Guava RateLimiter 风格令牌桶才能正确削峰。

3. AgentRouter：分类模型做意图识别

构造函数里把所有 Agent 的 id + description 拼进 system prompt，调用一个便宜模型（推荐 qwen-turbo）让它只输出一个 Agent id。这比自己写关键词表可扩展但不可控，推荐三层结构：

  1. 规则前置：正则/关键词命中（如"我的订单号"→客服）直接路由，不烧模型钱；
  2. 模型分类：兜底用小模型，输出 agentId；要求模型同时输出置信度，低于阈值走 fallback；
  3. Embedding 检索：把每个 Agent 的 description/示例问题向量化，用户问题向量做 top-K 检索（Agent 数量多、描述长时用）。

冷启动注册靠 Spring 依赖注入：new AgentRouter(builder, List<Agent> agents) 自动收集所有 @Component Agent 实现，无需手工注册。

4. AuditLog：为什么必须存 question+answer

只存调用元数据（谁用了什么 Agent、耗时多少）不够做质量闭环。必须记录：

字段作用生产注意
id / at主键、时间戳雪花 ID + UTC，便于分库分表
userId / tenantId用户/租户维度分析租户隔离分区（索引要带 tenantId）
agentId / modelName看哪个 Agent/模型被调计费、模型升级对比用
latencyMs / statusSLA 监控接 Prometheus 做 P95/P99
question / answerbadcase 回灌、人工标注、安全审计PII 脱敏（手机号/身份证/银行卡），合规留存周期

Demo 用 ConcurrentLinkedDeque<AuditRecord> + MAX_SIZE=1000 防止内存爆掉；生产换 Kafka + 落 OSS/ClickHouse，既不阻塞请求线程又能做离线分析。

核心 API 代码

下面是平台 Controller 主流程的最小可运行版（Demo 中同名类即为完整实现，此处抽出核心骨架方便理解）：

@RestController
@RequestMapping("/platform")
public class PlatformController {

    private final RequestGuard guard;
    private final AgentRouter router;
    private final AuditLog audit;

    public PlatformController(RequestGuard g, AgentRouter r, AuditLog a) {
        this.guard = g; this.router = r; this.audit = a;
    }

    @PostMapping("/ask")
    public ResponseEntity<?> ask(@RequestHeader("X-User-Id") String userId,
                                 @RequestBody Map<String,String> body) {
        // 1) 限流
        if (!guard.tryAcquire(userId)) {
            return ResponseEntity.status(429)
                .header("Retry-After", "60")
                .body(Map.of("error","rate_limited"));
        }
        String question = body.get("question");
        long start = System.currentTimeMillis();

        // 2) 路由 + 3) 执行
        Agent agent = router.route(question);             // 选 Agent
        Agent.AgentResponse resp = agent.handle(          // 执行业务
            new Agent.AgentRequest(userId, question));

        // 4) 审计
        audit.log(new AuditLog.Record(UUID.randomUUID().toString(),
            Instant.now(), userId, agent.id(), question,
            resp.answer(), System.currentTimeMillis() - start));

        return ResponseEntity.ok(Map.of(
            "success", true,
            "routedAgent", agent.id(),
            "latencyMs", resp.latencyMs(),
            "answer", resp.answer()));
    }
}

新 Agent 接入只需要：@Component class XxxAgent implements Agent { public String id(){return "xxx";} ... }，路由、限流、审计自动生效——这就是"平台"的价值。

关键参数与配置

限流阈值

PER_MINUTE_LIMIT = 20（每用户每分钟）。
调参时机：上线压测后按 QPS 配额、模型 TPM（token/min）反推；VIP 租户单独抬升。

分类模型选择

路由用便宜模型（qwen-turbo / qwen-plus 即可）。
仅输出 1 个 Agent id，token 消耗极低，成本 < 业务调用的 1%。

审计队列容量

Demo MAX_SIZE = 1000（内存 deque 上限）。
生产关闭内存上限，改异步落库/Kafka；设置 queue-capacity 背压。

超时隔离

Demo 未显式设超时。
生产每个 Agent 必须配置独立 ChatClient 超时（如 30s）+ 线程池隔离，避免慢 Agent 拖垮整个平台。

配置项默认值（Demo）生产建议
spring.ai.dashscope.chat.options.model（路由分类）与业务同模型固定为 qwen-turbo 节省成本
RequestGuard.PER_MINUTE_LIMIT20按用户等级/租户套餐差异化
AuditLog 存储内存 ConcurrentLinkedDequeKafka → ClickHouse / OSS，保留 90~180 天
Agent 超时未设（HTTP 默认）Resilience4j timeLimiter，每个 Agent 独立线程池
路由 fallback硬编码 learning Agent独立 fallback Agent（通用小模型 + 拒答策略）

边界与常见坑

  1. 限流窗口边界爆量：Demo 用"按分钟清 0"的固定窗口，在窗口交界会放过 2 倍流量。生产换 Sentinel 令牌桶或滑动窗口；全局、租户、用户三级配限额。
  2. 一个慢 Agent 拖垮整个平台：所有 Agent 共用默认 Tomcat 线程池，RAG/慢查询 Agent 会吃满线程。必须给每个 Agent 独立线程池 + @TimeLimiter 超时熔断，超时走降级话术。
  3. 审计日志 PII 泄漏：直接存 question/answer 很可能落手机号、身份证、地址。上线前必须加 PII 脱敏正则/模型，日志存储加密，按合规要求设留存周期（如金融 6 个月、教育 1 年）。
  4. 路由误判 = 用户体验事故：分类模型不是 100% 准确，一定要有 fallback Agent + 低置信度兜底规则；监控"路由→最终点赞/点踩"指标持续调优，误路由率 > 5% 就要加规则或微调分类器。
  5. 用户 prompt 没做安全过滤：平台入口必须做涉黄/涉政/暴力/Prompt 注入检测（阿里云绿网 + 百炼内容安全），否则用户一条注入就能拿到内部 Agent 的 system prompt 甚至打到下游工具。
  6. 没有 metrics 等于裸奔：必须打指标——总请求量、429 数、5xx 数、各 Agent 调用占比、P50/P95/P99 延迟、路由到 fallback 比例。接 Micrometer + Prometheus + Grafana。
  7. 灰度发布缺失：新 Agent 直接 100% 流量风险大。入口层加按 userId 哈希/白名单的灰度规则，新 Agent 先接 5% 流量观察。

生产集成方向

限流熔断 → Alibaba Sentinel

替换 RequestGuard：QPS 限流、热点参数限流（按 userId/tenantId）、熔断降级、集群限流流控，Dashboard 实时推规则。

动态配置 → Nacos

Agent 清单、路由规则、限流阈值、模型切换全部走 Nacos 配置中心热更新，不重启生效。

认证鉴权 → Spring Security

接 OAuth2 / JWT，X-User-Id 从 token 解析而非前端传；不同租户可访问的 Agent 清单做 RBAC 过滤。

多租户隔离

三层隔离：① 租户维度 Agent 清单；② 租户独立限流配额；③ 审计日志按租户分区；④ 模型 API Key 按租户计费。

可观测 → 百炼 Tongyi-Insight

接入阿里云百炼的 Trace/Tracing 能力，看每个 Agent 的 token 消耗、链路耗时、用户反馈评分。

网关入口 → Spring Cloud Gateway

把 /platform/** 放到 SCG，统一做鉴权、限流、日志染色、灰度路由；后面挂多个 Agent 平台实例。

成本设计

成本设计

路由用便宜模型，业务用贵模型。 分类只需要 1~2 个字符输出，用便宜模型省成本。

成本分层的典型占比（按一次请求）：

  • 路由分类：qwen-turbo，输入 ~100 token、输出 ~5 token，< ￥0.0001/次；
  • 业务 Agent：根据任务用 qwen-plus / qwen-max，输入 1k~10k token、输出 500~2k token，￥0.01~0.2/次；
  • Embedding/RAG：text-embedding-vX，均摊到每次约 ￥0.001；
  • 即路由成本占比 < 1%，但能节省"把闲聊问题误路由给 qwen-max"的浪费，路由层是最大的成本 ROI 点。

Q1：路由和业务用同一个大模型？

Q2：误路由率高时优先做？

Q3：下面哪个不是本课骨架已覆盖的能力？

课后问答

  • 如何做多租户？ — 在入口层解析 JWT 拿到 tenantId，AgentRouter 按租户过滤可见 Agent，RequestGuard 按 (tenantId, userId) 二级限流，AuditLog 按 tenantId 分区落库；Nacos 下发租户级配置。
  • 模型主备切换怎么实现？ — 在 Agent 内部包一层 ChatClient 代理：主模型（如 qwen-max）超时/报错时 fallback 到备模型（如 qwen-plus），并在 AuditLog 标 fallback=true 上报；同时接 Sentinel 熔断，连续失败 N 次自动切流到备模型 30s。
  • 新 Agent 如何安全上线？ — 三步走：① 注册后默认不进路由（Nacos 开关灰度）；② 按白名单 userId 或 5% 流量哈希灰度；③ 对比灰度组与对照组的"回答点踩率/平均延迟/token 成本"，指标不差再全量；回滚开关随时可切。
