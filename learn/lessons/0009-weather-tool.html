天气工具：外部 API 风格的 Tool

本课目标：把 Tool 从"数学函数"升级到"外部 API 风格"，学会结构化返回 + 显式 error 字段的设计。

配套 Demo：demos/10-weather-tool。参考卡：天气工具快速参考。

你今天要获得的一个能力

能独立设计一个"外部 API 风格"的 Spring AI Tool：用 @Tool + @ToolParam 暴露能力，用结构化 record（含 supported/error 字段）返回结果，让模型在多工具场景下按 description 正确选工具、组合结果、并在失败时给出友好话术。

教学取舍

Demo 用内置 Mock 数据，不调真实气象 API。真实项目里替换成 RestClient 即可，方法签名不变。

原理与机制：多工具场景下模型是怎么选工具的

从 0008 的 Calculator 单工具起步，真实业务里模型面前往往摆着一组工具：天气、交通、景点、餐厅、订单……模型并不会"看方法名"，它只看两件事：System Prompt 里的角色约束，以及每个 @Tool(description=...) / @ToolParam(description=...) 里写的自然语言描述。框架把所有工具签名序列化成一份 JSON Schema 发给模型，模型读完用户问题后自己决定：要不要调、调哪一个、参数填什么。

用户: "周末去哪玩？杭州和上海哪个合适？"
        │
        ▼
┌────────────────────────────────────────────┐
│ 模型读 tool schemas（JSON Schema 形式）    │
│  - currentWeather(desc="查实时天气...")    │
│  - forecast(desc="查未来3天预报...")       │
│  (未来) - searchAttraction / queryTraffic  │
└────────────────────────────────────────────┘
        │ 决定：先 forecast("hangzhou") 再 forecast("shanghai")
        ▼
┌────────────────────────────────────────────┐
│ Spring AI 回调 WeatherTools.forecast()     │
│ 返回 ForecastResult（结构化 record）       │
└────────────────────────────────────────────┘
        │ tool result 作为 assistant message 回传
        ▼
┌────────────────────────────────────────────┐
│ 模型组合两个城市的 forecast → 输出建议     │
└────────────────────────────────────────────┘

关键点有三。第一，选工具是模型的决定，所以 description 的质量直接决定正确率；"模糊提问"（"周末去哪玩"）会触发多次串行工具调用，而不是一次——模型拿到杭州预报后才决定要不要再查上海。第二，工具结果要结构化：如果工具返回一段中文长文本，模型需要再"读"这段文本抽字段，容易丢信息也浪费 token；返回 record/Map（被框架自动转成 JSON），模型相当于直接拿到一个对象，组合推理更稳。第三，错误也是结果：城市不支持不要抛异常（抛异常会让框架把 stacktrace 回传给模型，脏上下文），而是返回 supported=false + error 字段，模型读到就会礼貌告知用户支持哪些城市。

demo 09（calculator-tool）是单工具 + 纯函数场景，模型只能算对或算错；demo 10（本课）开始引入"外部 API 风格"的 mock 和显式错误字段；demo 11（order-agent）会进一步在多工具间加上下文（用户身份、订单状态），让工具结果根据会话上下文变化。这三个 demo 构成"函数 → API → 上下文 Agent"的三段爬坡。

设计要点

  • 返回 record + supported=false/reason，让模型据此说人话，而不是抛异常。
  • System Prompt 明确"必须调工具、不许口算"。
  • 参数命名与描述保持一致（city=英文小写拼音）。
  • 多个工具靠 @Tool(description) 做路由，description 要写清"什么场景用、参数格式、失败时返回什么"。

核心 API 与代码：WeatherTools 源码导读

下面是 demo 中 WeatherTools 的核心骨架（省略重复代码）。请重点看三件事：description 怎么写、参数怎么标注、错误怎么结构化返回。

@Component
public class WeatherTools {

    // Mock 数据：真实项目换成 RestClient / WebClient 调用外部 API
    private static final Map<String, Snapshot> CITY_MOCK = Map.of(
            "beijing",  new Snapshot("北京", 18, "多云", 45, 3),
            "shanghai", new Snapshot("上海", 22, "小雨", 78, 4),
            "hangzhou", new Snapshot("杭州", 24, "阴",  65, 2),
            "shenzhen", new Snapshot("深圳", 28, "晴",  55, 3)
    );

    @Tool(description = """
            查询指定城市当前天气。适用于用户询问"现在的天气/温度/湿度"等实时信息。
            city 需要使用城市英文小写拼音，例如 beijing / shanghai / hangzhou / shenzhen。
            如果城市未支持，会返回 supported=false 让上层告知用户。
            """)
    public CurrentWeather currentWeather(
            @ToolParam(description = "城市英文小写拼音，例如 beijing") String city) {
        Snapshot s = CITY_MOCK.get(city == null ? "" : city.toLowerCase());
        if (s == null) {
            // 关键：不要抛异常，返回带 error 的结构化对象
            return new CurrentWeather(false, city, null, null, null, null, null,
                    "unsupported city: " + city + "; supported=" + CITY_MOCK.keySet());
        }
        return new CurrentWeather(true, city, s.displayName(), s.temperatureC(),
                s.condition(), s.humidityPercent(), s.windLevel(), null);
    }

    // record 字段名要清晰，模型会直接看到 JSON key
    public record CurrentWeather(boolean supported, String cityCode, String cityName,
                                 Integer temperatureC, String condition,
                                 Integer humidityPercent, Integer windLevel,
                                 String error) {}
    public record ForecastResult(boolean supported, String cityCode,
                                 List<ForecastDay> days, String error) {}
    public record ForecastDay(String date, int minC, int maxC, String condition) {}
}

注册工具只需要一行 .defaultTools(tools)，Spring AI 会自动扫描 @Tool 注解并把方法签名转成模型能看懂的 JSON Schema：

this.chatClient = builder
    .defaultSystem("""
            你是一名"出行天气助手"。规则：
            1. 涉及具体城市天气/温度/出行建议时，必须先调用工具，禁止凭想象回答。
            2. 工具返回 supported=false 时，礼貌告知用户目前支持的城市列表。
            3. 最终回答使用中文，先给结论（是否适合出行），再给数据依据。
            """)
    .defaultTools(tools)   // <-- 注册所有 @Tool 方法
    .build();

为什么返回 Map/record 比返回自然语言好？ 框架会把工具返回值序列化成 JSON 塞进对话历史。结构化 JSON 的 token 数更可控、字段语义清晰（模型直接拿到 temperatureC=22 而不是要从"今天上海 22 度有小雨"里再抽数字），并且同一份结果可以被多个后续推理步骤复用（例如"周末去哪玩"要同时比较两个城市）。

工具结果回传的最佳实践

结构化优先

返回 record / Map / List，字段名用英文且语义明确（temperatureC、condition），避免长段中文。时间用 ISO 格式 yyyy-MM-dd。

错误也要结构化

不抛异常，返回形如 {supported:false, error:"city not found", available:["beijing","shanghai",...]}。模型能据此给出"暂不支持成都，目前支持北京/上海/杭州/深圳"这种友好话术。

描述要写"场景"

@Tool(description) 要写清"用户问什么的时候该用我"，这是多工具选路由的唯一依据。例如 forecast 的描述里要出现"未来几天/周末/出行安排"这些触发词。

参数要做归一化

模型可能传"北京市""北京""beijing""Beijing"——在方法体第一行统一 toLowerCase()、去"市/省"后缀、做别名映射（"魔都"→"shanghai"），再去查数据源。

动手

  1. 问"明后天去杭州合适吗？" → 模型应先调 forecast("hangzhou")。
  2. 问"成都天气如何？" → 应看到 supported=false 的话术。
  3. 把 currentWeather 加一个 timezone 字段并让模型使用。
  4. 追加一个"模糊提问"实验：问"周末想出去玩，北京和杭州哪个合适？"观察模型是否串行调用两次 forecast 再做比较。

关键参数与配置

@Tool(description)方法注解必填多工具路由的唯一依据，必须写清"什么问法触发我"。
@ToolParam(description)参数注解可选但建议参数含义/格式/示例有歧义时必写（例如 city="城市英文小写拼音"）。
spring.ai.dashscope.api-keyapplication.yml空通过环境变量 DASHSCOPE_API_KEY 注入，生产放配置中心（Nacos/ACM）。
ChatClient.defaultTools(tools)Builder 方法—注册工具；可用 tools(Object...) 在单次 prompt 级别覆盖/追加。
toolCallBack / returnDirect框架高级选项false需要"工具结果直接返回给用户不再经过模型"时使用（例如纯算术）。
超时设置RestClient/WebClient无默认调真实外部 API 必须配 connect/read timeout，建议 2-5s。

边界与常见坑

  1. 外部 API 必须加超时、熔断、重试：模型一次提问可能触发多次工具调用（"周末去哪玩"会并/串行调多次 forecast），一次慢查询会把整条链路拖到 30s 以上。生产请接入 Resilience4j（@TimeLimiter+@CircuitBreaker）或 Sentinel，超时降级为返回 {supported:false, error:"天气服务暂不可用"}，而不是抛异常。
  2. API Key 严禁写在 yml 里提交到 Git：spring.ai.dashscope.api-key 通过环境变量 / Nacos / 阿里云 KMS 注入；CI 里用 Secret；多环境（dev/staging/prod）分开 Key，避免配额被刷爆时全站受影响。
  3. 参数归一化一定在 Tool 方法里做：模型会传"北京""北京市""beijing""Beijing""首都"各种形态。仅靠 @ToolParam(description) 约束不够——要在方法入口 trim().toLowerCase()、建别名字典（"魔都"→"shanghai"）、去掉"市/省/区"后缀；未识别的不要返回 500，返回结构化错误让模型告诉用户。
  4. Mock 数据只用于教学，生产必须换真实 API + 缓存：天气这种分钟/小时级变化的数据适合 Caffeine/Redis 做 TTL=10min 的缓存，按 city 维度缓存，能把外部 QPS 压到原来的 1/100 以下。注意缓存 key 必须用归一化后的 city code。
  5. 不要在 Tool 里做阻塞 IO 占满 Tomcat 线程：当单次请求会触发 N 次工具调用时，务必用 WebClient 响应式调用，或把 Tool 方法标记为异步返回，否则高并发下很快打满 200 个 Tomcat 线程。
