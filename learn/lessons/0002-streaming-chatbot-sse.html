流式 ChatBot 与 SSE

本课的目标很小：把一次性返回的 ChatBot，改造成可以边生成边返回的流式接口。

配套 Demo：demos/03-streaming-chatbot。参考卡片：流式 ChatBot 与 SSE 快速参考。

你今天要获得的一个能力

能力目标：看到一个流式 ChatBot 接口时，能说清楚它如何用 ChatClient.stream() 和 HTTP SSE 把模型生成过程逐步返回给客户端。

上一课你已经掌握了同步调用：

chatClient.prompt()
        .user(question)
        .call()
        .content();

这一课只替换一个核心动作：

chatClient.prompt()
        .user(question)
        .stream()
        .content();

为什么 ChatBot 需要流式输出？

如果模型回答很长，同步接口会让用户一直等待，直到完整答案生成完毕。流式输出可以让用户更早看到内容，体验接近“助手正在打字”。

同步回答

适合短问题、后台任务、环境验证。缺点是长回答时用户一直等待。

流式回答

适合智能客服、代码解释、文档总结。模型生成一部分，用户就能看到一部分。

核心链路

HTTP POST /chat/stream
  ↓
Spring Controller
  ↓
ChatClient.prompt()
  ↓
.user(question)
  ↓
.stream()
  ↓
.content()
  ↓
Flux<String>
  ↓
text/event-stream

这里的新概念有两个：

  • Flux<String>：表示一串未来会陆续到达的字符串片段。
  • text/event-stream：HTTP SSE 的响应类型，表示服务端会持续向客户端发送事件。

SSE 是什么？

SSE 是 Server-Sent Events。它适合“服务端持续向客户端推送文本”的场景。

为什么本课用 SSE，而不是 WebSocket？

因为 ChatBot 流式回答主要是服务端向客户端单向输出。SSE 更简单，和普通 HTTP 更接近，适合作为后端流式输出的第一课。

SSE 原理：为什么"打字机效果"不是 WebSocket

SSE（Server-Sent Events）本质上是一个 不关闭的 HTTP 长连接：客户端发起一次普通 HTTP 请求，服务端把 Content-Type 设为 text/event-stream，然后持续往响应体里写 data: ... 格式的文本帧，直到写完一个特殊的 data:[DONE] 帧表示结束。整个过程走的是标准 HTTP，不需要协议升级，不需要双向帧，代理、鉴权、Cookie 全部沿用现成方案。

下面是一次流式回答从模型到浏览器的完整链路。注意：响应式流的"背压"是从 Flux 一路传递到 Servlet/WebFlux 容器的，所以慢客户端不会让服务端 OOM。

┌─────────┐  token chunk  ┌──────────────────┐  onNext  ┌────────────┐  write+flush ┌──────────┐
│ 大模型  │ ────────────▶ │ Spring AI        │ ───────▶ │ WebFlux /  │ ───────────▶ │ TCP 连接 │
│(DashScope│  Server-Sent  │ ChatClient.stream│ Flux<String>| Servlet 3.1|   text/event-│  浏览器    │
│  流式响应)│  Events over │   .content()     │          │ SseEmitter │   stream 帧  │ EventSource│
└─────────┘   HTTPS       └──────────────────┘          └────────────┘              └──────────┘
              ▲                                                            ▲
              │ 1 条 HTTP POST，不升级协议                                    │ data:xxx\n\n 逐帧
              │ Content-Type: text/event-stream                            │ 以 \n\n 分隔
              │ Cache-Control: no-cache                                    │ 末帧 data:[DONE]

SSE 帧长什么样

每一帧以一行 data: 开头，以两个换行 \n\n 结束。浏览器 EventSource 收到空行就把这一帧交给 onmessage。模型每吐出一个（或一小段）token，服务端就写一帧。

data:你

data:好

data:，

data:我是

data:通义千问

data:[DONE]

SSE

HTTP 单向（服务端→客户端），基于 HTTP/1.1 长连接或 HTTP/2 多路复用；文本帧，浏览器原生 EventSource API；自动重连；穿透企业代理最容易；最适合 ChatBot 这类"问一句、答一段"场景。

WebSocket

双向全双工，需要 HTTP Upgrade 握手机制；二进制/文本帧皆可；要自己做心跳、重连、子协议；适合双向高频交互（协同编辑、在线游戏、语音双向流），对单向 LLM 输出属于"杀鸡用牛刀"。

curl 里 -N 和 Accept: text/event-stream 分别干嘛

  • curl -N（--no-buffer）：禁用 curl 自己的 16KB 输出缓冲。不加 -N 你会看到所有 token 攒成一坨在流结束时一次性打印，误以为服务端没流。
  • -H 'Accept: text/event-stream'：告诉 Spring MVC 你想要 SSE 渲染器。Spring 的内容协商会根据 produces + Accept 选择 text/event-stream 的 HttpMessageConverter，否则可能回退成 JSON 序列化整个 Flux。
  • 注意：SSE 是 POST 也可以（本课的 /chat/stream 就是 POST），不是只能 GET。EventSource 原生只支持 GET，POST 场景用 fetch + ReadableStream（见下一节浏览器示例）。

完整可运行版本（含 DTO、异常兜底、日志）

下面是贴近企业写法的 Controller，关键点都加了注释：

@RestController
@RequestMapping("/chat")
public class StreamingChatController {

    private final ChatClient chatClient;

    // 推荐构造器注入；ChatClient 通过 ChatClient.Builder 构建，
    // Spring AI Alibaba starter 会自动把 DashScope 的 ChatModel 注入进来
    public StreamingChatController(ChatClient.Builder builder) {
        this.chatClient = builder.build();
    }

    // 请求体 DTO，Java 17+ 直接用 record
    public record AskRequest(String question) {}

    /**
     * 流式问答接口
     * - produces = TEXT_EVENT_STREAM_VALUE 告诉 Spring 用 SSE 渲染
     * - 返回 Flux<String> 即可，框架自动把每个 onNext 写成一个 data: 帧
     * - 流结束时 Spring 会自动写 [DONE] 吗？不会，真正的 [DONE] 是前端约定；
     *   与 OpenAI 兼容时通常自己 concat 一个 "[DONE]" 哨兵（见下）
     */
    @PostMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> stream(@RequestBody AskRequest req) {
        return chatClient.prompt()
                .user(req.question())
                .stream()
                .content()
                // 生产环境必须兜底：模型侧异常如果不 resume，
                // 会以 error 信号关闭连接，前端只看到连接中断拿不到原因
                .onErrorResume(e -> Flux.just("[ERROR] " + e.getMessage()));
    }
}

浏览器端消费 SSE：10 行 JS 实现打字机

后端是 SSE，前端最直接的消费方式是 EventSource。但 EventSource 只支持 GET，所以当你需要 POST JSON（带问题体）时，用 fetch + ReadableStream 即可。下面给出一个最小可跑的前端片段，直接粘到浏览器 DevTools Console 或前端工程里都可以：

// POST + SSE 的最小前端示例
const output = document.body; // 实际项目换成 <pre> 或聊天窗口 DOM
const resp = await fetch('/chat/stream', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'Accept': 'text/event-stream' },
  body: JSON.stringify({ question: '用一句话解释 SSE' })
});

const reader = resp.body.getReader();
const decoder = new TextDecoder();
let buffer = '';

while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  buffer += decoder.decode(value, { stream: true });

  // SSE 帧以 \n\n 分隔，按帧切分
  let idx;
  while ((idx = buffer.indexOf('\n\n')) >= 0) {
    const frame = buffer.slice(0, idx);           // 形如 "data:你好"
    buffer = buffer.slice(idx + 2);
    const payload = frame.replace(/^data:\s?/, '');
    if (payload === '[DONE]') break;              // 后端约定的结束哨兵
    if (payload.startsWith('[ERROR]')) { console.error(payload); break; }
    output.append(payload);                       // 逐字追加到 DOM = 打字机效果
  }
}

如果你用 EventSource（GET 场景，比如服务端单向通知），代码更短：const es = new EventSource('/chat/stream'); es.onmessage = e => console.log(e.data);。

关键参数与配置

流式接口看起来只有一行 .stream().content()，但企业落地时你会频繁触碰下面这些配置/算子。

项默认值 / 常见取值什么时候需要调
spring.ai.dashscope.chat.options.temperature 模型默认（通义通常 0.7 左右） 客服/知识库场景建议 0~0.3 降低幻觉；创意写作 0.7~1.0。
produces = TEXT_EVENT_STREAM_VALUE 必须显式写 不写 Spring 会把 Flux<String> 当 JSON 数组一次性序列化，流不起来。
chunk 粒度（token 级 vs 句级） Spring AI 默认是模型侧原生 chunk，通常 token 级 想减少前端重绘/降低 chunk 数量：在 Flux 上加 .bufferTimeout(20, Duration.ofMillis(100)) 做句级聚合；不要用 delayElements，那是人为加延迟，会让首字更慢。
SSE 结束哨兵 无默认（Spring 不会自动发 [DONE]） 要对接 OpenAI SDK/前端统一协议时，在流末尾 .concatWithValues("[DONE]")。
响应超时 / 空闲超时 依容器（Tomcat 默认 connectionTimeout 20s；Netty 无默认） 长回答场景要在网关上把 proxy_read_timeout、应用端 server.netty.idle-timeout 调大到 60~120s，否则回答到一半被切断。
@CrossOrigin / CORS 默认不允许跨域 前端单独部署（不同域/端口）时必须配，否则浏览器预检直接拦。

边界与常见坑

踩坑 1：Nginx/网关默认缓冲吃掉"打字机"效果。

Nginx 反向代理默认开启 proxy_buffering on，会把后端的 SSE 帧攒满缓冲再吐给客户端，现象是等很久一次性返回全部内容。必须在 location 里加：

location /chat/ {
    proxy_pass http://backend;
    proxy_buffering off;          # 关键：关缓冲
    proxy_cache off;
    proxy_read_timeout 300s;      # 长回答必须调大
    proxy_set_header Connection '';  # SSE 走 HTTP/1.1 长连接，不要升级
    chunked_transfer_encoding off;
}

其他网关（Spring Cloud Gateway、APISIX、Ingress Nginx）同理，都有"响应缓冲"开关，要关掉。

踩坑 2：Flux 的 onError 信号会直接终止流，用户看到半截回答。

模型限流、网络抖动、单 token 序列化失败都可能触发 error。必须在 Flux 链末尾挂 onErrorResume，把错误转成一帧可读提示（例如 [ERROR] 模型暂时不可用），而不是让连接粗暴断开。前端也要对 EventSource 的 error 事件和 fetch reader.read() 异常做兜底。

踩坑 3：移动端网络切换 / 切后台不会自动续传。

SSE 基于 TCP 长连接，手机从 Wi-Fi 切 4G、锁屏、App 切后台，连接都会断。浏览器 EventSource 虽然有自动重连，但重连后之前已生成的 token 全丢了，用户体验就是回答从头开始/被截断。生产环境需要：(1) 后端给每次流分配 requestId，已生成内容落缓存；(2) 前端重连时带上 Last-Event-ID 或自定义 offset，后端从断点继续推送，或直接把已生成部分作为一次性历史补齐。

踩坑 4：别用 .delayElements() 控制速度。

delayElements 是给每个元素人为增加固定延迟，会让首字时间（TTFT）跟着变糟，纯属反模式。想"节流以降低前端重绘压力"，请用 .bufferTimeout(N, Duration) 或 .sample(Duration)，按时间/条数聚合 chunk 再发。
