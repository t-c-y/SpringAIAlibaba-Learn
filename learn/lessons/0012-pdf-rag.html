PDF 知识库：按页引用与上传灌库

本课目标：会用 PagePdfDocumentReader 处理 PDF，让回答能标注 [page N] 引用。

配套 Demo：demos/13-pdf-rag。参考卡：PDF RAG 快速参考。

你今天要获得的一个能力：能把任意业务 PDF（合同、手册、规范、研报）灌进向量库，让模型回答时强制以 [page N] 标注出处页码，做到回答可溯源、可审计，而不是"凭空编一段像真的一样的答案"。

PDF RAG 工作原理

和 Markdown/纯文本 RAG 的唯一本质差别在 DocumentReader 这一步。PDF 是二进制版式文件，自带页、字体、坐标、图片、表格，必须由专门的 Reader 负责"抽文本 + 保留页码"，后面的 Splitter、Embedding、VectorStore、Retriever 链路和 Markdown RAG 完全一致。

                ┌──────────────────────────────────────────────────────────┐
                │                      PDF 文件（二进制）                  │
                └──────────────┬───────────────────────────────────────────┘
                               │  PagePdfDocumentReader
                               ▼
                ┌──────────────────────────────────────────────────────────┐
                │  List<Document> 一页一个 Document                         │
                │  metadata: { page_number=1, file_name="x.pdf" }          │
                │  metadata: { page_number=2, file_name="x.pdf" }          │
                └──────────────┬───────────────────────────────────────────┘
                               │  TokenTextSplitter(400, 200)
                               ▼
                ┌──────────────────────────────────────────────────────────┐
                │  子 chunks（继承 page_number metadata）                  │
                └──────────────┬───────────────────────────────────────────┘
                               │  vectorStore.add(...) → Embedding
                               ▼
                ┌──────────────────────────────────────────────────────────┐
                │  VectorStore（向量 + 原文 + metadata）                    │
                └──────────────┬───────────────────────────────────────────┘
                               │  question → similarity search topK
                               ▼
                ┌──────────────────────────────────────────────────────────┐
                │  召回 topK chunks（带 page_number）→ 拼进 Prompt          │
                │  System: "回答必须以 [page N] 标注引用页码"                │
                               ▼
                         LLM 生成带引用的回答

第一步 Reader：Spring AI 提供 PagePdfDocumentReader（按页切，每个 Document 对应一页，metadata 带 page_number，最适合做"按页引用"）和 ParagraphPdfDocumentReader（按视觉段落自动聚合，段落可能跨页，会丢页码精度）。底层用 Apache PDFBox 解析 PDF 算子流抽文字；扫描版 PDF（整页是一张图）抽不到任何文字，必须先 OCR。

第二步 Splitter：单页 PDF 往往超 Embedding 上下文，需要再次切。常用 TokenTextSplitter(400, 200)——每个 chunk 400 token，相邻 chunk 重叠 200 token 以避免句子/表格被硬切断；切出来的子 chunk 会继承原 Document 的 metadata，所以 page_number 不会丢。

第三步 Retrieval + Prompt 约束：召回的 chunks 把 page_number 一起送进 Prompt；System Prompt 里必须写死"回答中凡引用资料的部分都要以 [page N] 标注页码，N 从 1 开始"——这是让模型不编造出处的关键。没有这句硬约束，模型大概率会省略引用或瞎编页码。

关键差异

  • 相比 TextReader，PagePdfDocumentReader 会在 Document metadata 上写入 page_number。
  • System Prompt 里可要求模型在回答末尾用 [page N] 引用来源，实现可追溯。
  • 扫描版 PDF 需要先 OCR，本课不覆盖。

Reader 选型对比

PagePdfDocumentReader

一页一个 Document，metadata 必带 page_number。本课选用，因为要做 [page N] 引用。

ParagraphPdfDocumentReader

按视觉段落聚合，段落可能跨页，页码不精确。适合"不需要引用页码、只做语义召回"的场景。

自写 Apache PDFBox

自己 new PDDocument + PDFTextStripper 抽文本，灵活但要自己处理坐标、表格、CJK 字体，成本高。

核心 API / 完整 ingest 代码

PDF RAG 和 Markdown RAG 的代码骨架完全一致，唯一区别就是 Reader 换成 PagePdfDocumentReader。下面是 /pdf/upload 接口的典型实现：

@RestController
@RequestMapping("/pdf")
public class PdfRagController {

    private final VectorStore vectorStore;
    private final EmbeddingModel embeddingModel;
    private final ChatClient chatClient;

    // 强制引用页码的 System Prompt——这是避免模型瞎编出处的关键
    private static final String SYSTEM_PROMPT = """
        你是企业文档助手，必须严格基于下方"参考资料"回答。
        规则：
        1. 答案只能来自参考资料，资料里没有的就说"文档中未提及"；
        2. 每个事实性陈述后必须用 [page N] 标注引用页码，N 从 1 开始；
        3. 如果一句话同时来自多页，写成 [page 3,5]；
        4. 禁止编造页码，禁止引用参考资料里没有的内容。
        参考资料：
        {documents}
        """;

    // 文件上传接口：MultipartFile → 临时文件 → Reader → Splitter → VectorStore
    @PostMapping("/upload")
    public Map<?, ?> upload(@RequestParam("file") MultipartFile file) throws Exception {
        // 1. 类型/大小校验（必须做，见边界坑）
        String original = file.getOriginalFilename();
        if (original == null || !original.toLowerCase().endsWith(".pdf")) {
            return Map.of("success", false, "error", "仅支持 PDF");
        }
        if (file.getSize() > 20 * 1024 * 1024) {
            return Map.of("success", false, "error", "PDF 不能超过 20MB");
        }

        // 2. MultipartFile 转临时文件（PagePdfDocumentReader 接受 Resource）
        Path tmp = Files.createTempFile("upload-", ".pdf");
        file.transferTo(tmp);

        // 3. 按页读 PDF：withPagesPerDocument(1) → 一页一个 Document
        PagePdfDocumentReader reader = new PagePdfDocumentReader(
                new FileSystemResource(tmp),
                PdfDocumentReaderConfig.builder()
                        .withPagesPerDocument(1)
                        .build());
        List<Document> docs = reader.get();

        // 4. 补业务 metadata（后续按文件名做隔离/删除/版本管理会用到）
        docs.forEach(d -> {
            d.getMetadata().put("file_name", original);
            d.getMetadata().put("upload_at", Instant.now().toString());
        });

        // 5. TokenTextSplitter 按 400 token 切、重叠 200——子 chunk 自动继承 page_number
        TokenTextSplitter splitter = new TokenTextSplitter(400, 200, 5, 10000, true);
        List<Document> chunks = splitter.apply(docs);

        // 6. 灌库（内部会调用 EmbeddingModel 向量化）
        vectorStore.add(chunks);

        Files.deleteIfExists(tmp);
        return Map.of("success", true, "fileName", original,
                      "pages", docs.size(), "chunks", chunks.size());
    }

    @PostMapping("/ask")
    public Map<?, ?> ask(@RequestBody Map<String,String> req) {
        String question = req.get("question");
        // 7. 检索时按 page_number 聚合也可以，这里走标准 similarity search
        List<Document> hits = vectorStore.similaritySearch(
                SearchRequest.query(question).withTopK(5));
        String docs = hits.stream()
                .map(d -> String.format("[page %s] %s",
                        d.getMetadata().get("page_number"),
                        d.getText()))
                .collect(Collectors.joining("\n\n"));

        String answer = chatClient.prompt()
                .system(s -> s.text(SYSTEM_PROMPT).param("documents", docs))
                .user(question)
                .call().content();
        return Map.of("success", true, "answer", answer);
    }
}

关键点：TokenTextSplitter 切出来的子 chunk 会原样继承父 Document 的 metadata，所以即使一页被切成 3 片，每片都带正确的 page_number——这是页码引用能成立的基础。

关键参数与配置

参数 / 配置项默认何时调 / 说明
PdfDocumentReaderConfig.withPagesPerDocument(n)1设为 1 才能一页一个 Document、引用页码精确；手册/合同建议保持 1，长篇论文可设 2-3 减少 chunk 数。
TokenTextSplitter(defaultChunkSize, ...)800PDF 场景建议 400——一页 A4 中文约 500-800 字，400 token 能保证单 chunk 落在 Embedding 最佳区间，且单页信息可被多次召回。
TokenTextSplitter(..., minChunkSizeChars, maxNumChunks, keepSeparator)350, 10000, truekeepSeparator=true 保留换行/句号，避免把句子切残。
SearchRequest.withTopK(k)4PDF 推荐 5-8——单页切多片后 topK=4 容易只命中同一页的相邻片，拉高 k 增加页码覆盖面。
spring.ai.vectorstore.simple.*（SimpleVectorStore）内存重启就丢，仅开发用。生产必须切 Redis/Milvus/OpenSearch/PGVector 等持久化实现。
上传文件大小限制（spring.servlet.multipart.max-file-size）1MBPDF 场景至少调到 20-50MB，并在 Controller 再做一次业务校验。

边界与常见坑（企业踩过的）：

  1. 扫描版 PDF 抽不到文字。PagePdfDocumentReader 底层是 PDFBox 抽文字算子流，扫描件整页是张图，返回的 Document text 会是空或乱码。必须先 OCR：可用 PaddleOCR/TrOCR/Azure Document Intelligence/阿里云文档智能，把 OCR 结果拼装成"伪页文本"再交给 Reader。不要等用户投诉"为什么上传完问什么都答不出"才排查。
  2. 表格跨页会被切断 + 抽不出结构。PDFBox 按文本坐标抽，跨页表格会被撕成两半，且行列对齐信息丢失。对财务/报表类 PDF，需要用 Camelot/Tabula（基于规则）或 LayoutLM/Azure Doc Intelligence（基于模型）先做表格结构化，再决定如何入向量库。
  3. 加密 PDF 直接报错。PDFBox 对加密文档会抛 InvalidPasswordException。在 Controller 入口要 catch 住，返回明确提示"该 PDF 已加密，请先解密再上传"；已知密码的场景可用 document.decrypt(password) 后再读。
  4. [page N] 页码从 1 开始，不是 0。Spring AI 的 page_number 是从 1 开始写的（对应用户看到的 PDF 页码），System Prompt 必须写"N 从 1 开始"，否则模型有时会自己减 1 导致页码错位。
  5. SimpleVectorStore 重启即失忆。它是纯内存实现，服务一重启用户上传过的 PDF 就全没了，上传接口会"看起来正常"但回答全是"文档未提及"。生产必须接入持久化 VectorStore（Redis/Milvus/PGVector/OpenSearch），不要把 SimpleVectorStore 上预发/生产。
  6. resources 下的 sample.pdf 不存在会启动失败。如果写了 new ClassPathResource("sample.pdf") 做启动时自动灌库，一定要确保 resources 目录真的放了这个文件，否则启动直接抛 FileNotFoundException；建议用 @ConditionalOnResource 或 resource.exists() 判断，缺文件就跳过灌库而不是挂掉。
  7. 上传接口必须做大小 + 类型校验。不能只靠前端，Controller 必须二次校验扩展名/Magic Number（PDF 头是 %PDF）和大小，否则有人上传 2GB 伪 PDF 会直接把内存/磁盘打爆。

课后问答

  • "能不能按章节切分？" —— 可以，但不是开箱即用。需要先用 PDFBox 解析书签（catalog.getDocumentCatalog().getDocumentOutline()）或基于字号/正则识别章节标题，再按章节边界合并 chunk；或先用一个 LLM 调用从目录页提取章节→页码映射，再按页码区间二次聚合。
  • "如何做同 PDF 的版本管理？" —— 推荐在 metadata 里加 doc_id（文件内容 hash）+ version；上传新版本时先按旧 doc_id 调 vectorStore.delete(FilterExpression) 删除旧 chunks，再灌新的。SimpleVectorStore 不支持按 filter delete，生产级 VectorStore 才支持。
  • "扫描版合同一定要 OCR，有没有轻量方案？" —— 阿里云文档智能（DocumentMind）/ 腾讯云 OCR 都是 API 接入，不用自己部署模型；本地可用 PaddleOCR（中文效果最好）跑批。OCR 完后把每一页识别出的文字封装成带 page_number 的 Document，下游链路完全复用即可。
