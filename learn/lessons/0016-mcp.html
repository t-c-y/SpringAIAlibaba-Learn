MCP 工具集成：跨进程工具的动态发现

本课目标：理解 MCP 思想——工具通过标准协议**动态发现与调用**，而不是在编译期用 @Tool 注册。

配套 Demo：demos/17-mcp-integration。参考卡：MCP 快速参考。

你今天要获得的一个能力：把 Tool Calling 从单进程 @Tool 方法，扩展为跨进程、跨语言、可动态发现的 MCP 工具服务——任何团队用任何语言写的能力，只要实现 MCP 协议，就能被你的 Spring AI Agent 像本地 @Tool 一样直接调用。

MCP 协议基础

MCP（Model Context Protocol） 是由 Anthropic 发起、现在多家厂商共同推动的开放协议，目标是让"给 LLM 接工具/上下文"这件事从各自私有 SDK，变成像 HTTP 一样的标准。

MCP 消息层使用 JSON-RPC 2.0（request/response/notification 三类消息），传输层支持三种：

传输方式适用场景特点
stdio本地子进程（如本地 IDE 插件、CLI 工具）Client fork 子进程，通过 stdin/stdout 走 JSON-RPC；零端口、零网络
SSE远程 HTTP 服务Server 推流，Client 用 POST 发请求、SSE 接响应；长连接
Streamable HTTP远程 HTTP 服务（新版）在普通 HTTP 上支持流式响应，比 SSE 更易过网关/负载均衡

MCP 暴露两类核心能力：

  • Tools（函数调用）：Server 声明一组可被 LLM 调用的工具，带 inputSchema（JSON Schema）；和你在 demo 09/10/11 用的 @Tool 语义等价，但跨进程。
  • Resources（上下文资源）：暴露可被 LLM 读取的资源（本地文件、数据库、API 返回的结构化数据），用 URI 寻址，类似"给模型一个只读文件系统"。

本课只讲 Tools；Resources 留到后续课程的 RAG/知识库场景再讲。

MCP vs Function Calling：从单进程到跨服务

维度Function Calling（@Tool）MCP Tools
调用方式同 JVM 内反射调用 @Tool 方法Client/Server 架构，走 JSON-RPC over stdio/SSE/HTTP
部署边界工具代码必须和 Agent 在一个进程里跨进程、跨机器、跨语言（Python/Go/Node 写的工具都能调）
注册方式编译期/启动期 @Tool 扫描运行期 listTools() 动态发现，Server 可热插拔
复用粒度每个 Agent 自己配一套一个 MCP Server 可被多个 Agent/应用同时复用
安全/治理JVM 内方法，靠代码规范跨网络，必须做认证、鉴权、超时、熔断、审计
LLM 侧语义完全相同——模型都只是看到"工具名 + 描述 + inputSchema"，然后决定调不调、传什么参数

一句话总结：MCP 把 Tool Calling 从"本地方法调用"变成了"标准化的 RPC 调用"，LLM 看不到区别，但工程边界被彻底打开了。

两段式工具交互：list → call

MCP Tools 的核心只有两个动作，理解这两步就理解了整个协议。

 ┌─────────┐                                  ┌─────────────┐
 │  Agent  │                                  │ MCP Server  │
 │ (Client)│                                  │  (Tools)    │
 └────┬────┘                                  └──────┬──────┘
      │  1) tools/list                              │
      │  ─────────────────────────────────────────► │
      │                                             │  扫描本地/远程
      │                                             │  所有已注册工具
      │  2) [{name,description,inputSchema}, ...]   │
      │  ◄───────────────────────────────────────── │
      │                                             │
      │  (Agent 把工具清单拼进 System Prompt，       │
      │   交给 LLM 决策；每轮都可能触发调用)          │
      │                                             │
      │  3) tools/call {name, arguments}            │
      │  ─────────────────────────────────────────► │
      │                                             │  执行真实逻辑
      │                                             │  (读文件/查库/调API)
      │  4) {content: [...], isError: bool}         │
      │  ◄───────────────────────────────────────── │
      │                                             │
      │  (把结果回填给 LLM → 继续推理 → 要么再调工具，│
      │   要么输出 final answer)                     │
      └─────────────────────────────────────────────┘
  1. listTools()：Agent 启动时（或按需）向 Server 请求工具清单，Server 返回每个工具的 name、description、inputSchema（JSON Schema）——这三项就是 LLM 决定是否调用它的全部依据。
  2. callTool(name, args)：LLM 在推理中决定调用某个工具，Agent 把工具名 + 参数发给 Server，Server 执行真实逻辑（读文件、查数据库、调外部 API……）并把结构化结果返回。
  3. 循环：调用结果回填给 LLM，LLM 判断"还需要调别的工具吗"，需要则继续 callTool，不需要则输出最终答案。这个 loop 和 demo 09/10/11 的 in-process Tool Calling 完全同构，只是"工具执行"这一步从本地反射变成了 RPC。

教学取舍：MockMcpServer 说明

本课 Demo 使用一个教学版 REST 模拟——MockMcpServer：

  • GET /mcp/tools 返回工具清单（对应 tools/list）
  • POST /mcp/call 按 toolName + arguments 分发执行（对应 tools/call）

真实 MCP 是 JSON-RPC 2.0 over stdio/SSE/Streamable HTTP，消息格式不是 REST JSON 而是 JSON-RPC（带 id/method/params）。但交互语义完全一致：list 拉清单看 schema → call 按名传参 → 循环直到 final answer。先用 REST 版把这三个动作看透，再切真实 MCP SDK 时你只需要换传输层。

核心代码：McpAgentController 的 Agent Loop

Demo 的 McpAgentController 自己手写了一个最小 ReAct loop——这能让你看清 Agent 到底在循环什么（而不是被框架魔法遮住）。

// 伪代码：Demo 中 McpAgentService 的核心循环
List<ToolSpec> tools = mcpClient.listTools();           // ① 启动时拉工具清单
String systemPrompt = buildSystemPrompt(tools);         // ② 把 name/description/inputSchema 拼进 Prompt
List<Message> messages = new ArrayList<>();
messages.add(systemPrompt);
messages.add(new UserMessage(question));

for (int step = 0; step < MAX_STEPS; step++) {          // ③ 最多 N 步，防止死循环
    ChatResponse resp = chatClient.prompt(messages).call().chatResponse();
    String text = resp.getResult().getOutput().getText();

    if (isFinalAnswer(text)) {                          // ④ LLM 认为可以答了 → 输出
        return parseFinal(text);
    }
    ToolCall call = parseToolCall(text);                // ⑤ 否则解析出 toolName + args
    ToolResult result = mcpClient.call(call.name(), call.args()); // ⑥ 调 MCP Server
    messages.add(new AssistantMessage(text));
    messages.add(new ToolResultMessage(result));        // ⑦ 结果回填 → 继续下一轮
}
throw new IllegalStateException("超过最大步数");

这其实就是 LangChain ReAct Agent 的最小内核：Thought → Action → Observation → Thought → ... → Answer。区别只是 Action 阶段调的不是本地反射，而是跨进程的 MCP 调用。生产环境你不会手写这个 loop，而是直接把 MCP 工具注册进 ChatClient，由 Spring AI 框架替你循环；但先看懂手写版，出了问题你才知道 loop 卡在哪一步。

未来 Spring AI Alibaba 的 McpClient 使用方式（概念）

Spring AI 官方已提供 spring-ai-mcp-client 模块，Spring AI Alibaba 也会对齐这一套。一旦用官方 SDK，你就不用自己写 REST 调 /mcp/tools 和 /mcp/call，代码会和用 @Tool 几乎一样：

// 概念代码：以官方 spring-ai-mcp-client 发布的 API 形态为准
// 1) 指定传输方式连到 MCP Server（本地子进程 / 远程 SSE）
McpTransport transport = new SseMcpTransport("https://mcp.internal.com/sse");
// 或 stdio：new StdioMcpTransport("npx", List.of("-y", "@some/mcp-server"));

McpClient mcpClient = McpClient.create(transport);
mcpClient.connect();

// 2) 把 MCP Server 暴露的工具直接塞进 ChatClient，和 @Tool 同等地位
List<ToolCallback> mcpTools = mcpClient.listToolCallbacks();

ChatClient chatClient = chatClientBuilder
    .defaultTools(mcpTools)            // 就像 .defaultTools(new MyLocalTools())
    .build();

// 3) 正常调用，框架自动完成 list → 多轮 call → final answer
String answer = chatClient.prompt("帮我查一下库存并下单").call().content();

注意三点：(a) listToolCallbacks() 把 MCP 工具适配成了 Spring AI 标准的 ToolCallback，所以你可以和本地 @Tool 方法混着注册；(b) 多轮 loop 由框架自动执行，你只要在 builder 里配置 maxToolCalls 等参数；(c) transport 切换对上层完全透明——今天连本地 stdio，明天换远程 SSE，业务代码零改动。

关键参数与配置

Agent 最大轮次 max-steps

默认：Demo 中 MAX_STEPS=10。生产视场景设 5-20。过低会让复杂任务中途被切，过高会烧 token + 卡延迟。必须有，否则模型钻牛角尖会无限循环。

MCP 调用超时 call-timeout

默认：无（依赖 HTTP/RPC 默认）。一定要显式设（HTTP 客户端 connect/read timeout、未来 SDK 的 requestTimeout）。外部工具卡 30s 会把整个 Agent 链路拖死。

工具过滤 tool-filter

默认：listTools 全量注册。企业里一个 MCP Server 可能暴露 50+ 工具，全塞 Prompt 会让模型注意力稀释 + token 爆炸。按业务场景/用户角色做白名单过滤。

MCP Server 地址 transport

Demo：http://localhost:8096（REST 模拟）。生产通过配置中心下发，支持 stdio（本地子进程命令）、SSE URL、Streamable HTTP URL。建议用环境变量区分 dev/staging/prod。

认证 auth

默认：无（本地/内网）。跨网络必须带——API Key 放 Header、或 JWT/OAuth2。spring-ai-mcp-client 支持通过 ClientRequestObservationConvention / 自定义 interceptor 注入。

模型温度 temperature

Agent 场景建议 0–0.3。工具调用参数对格式敏感（JSON、枚举、数字），温度高了会出幻觉参数，MCP Server 侧要做参数校验兜底。

边界与常见坑

  1. MCP Server 绝不能信任 Client 传的任意参数。和本地 @Tool 一样，所有 string/number/enum 入参必须在 Server 侧按业务规则校验（长度、范围、权限），不能因为"是 Agent 传的"就放松。跨网络尤其要防注入——SQL、命令、路径穿越都要过滤。
  2. 工具清单过大会让 Prompt 爆炸。每个工具的 description + inputSchema 都会拼进 System Prompt。一个 Server 暴露 100 个工具是灾难——token 成本飙升、模型选错工具概率大增。必须按场景分组，或在 Client 侧做"先做工具路由，再加载工具集"的两层发现。
  3. 跨网络调用必须设超时/重试/熔断/审计。本地 @Tool 失败只是抛异常；远程 MCP 失败会带来网络分区、慢响应、雪崩。给每个工具调用设独立超时（建议 3-10s）、有限重试（仅幂等工具）、熔断（连续失败快速降级），并把每一次 call 的 (toolName, argsHash, caller, latency, ok) 落审计日志。
  4. 工具 schema 要版本化。MCP 工具的 name/inputSchema 是 LLM 决策的依据，改了参数就等于改了"Prompt 契约"。建议加版本（search_docs_v2）或在 Server 端做 schema 向后兼容，老 Agent 不会突然挂。
  5. 必须做权限认证与租户隔离。MCP Server 一旦暴露到网络，就等于你的内部能力"上架应用商店"。至少要有 API Key/JWT，按 caller 做工具级 ACL（不是所有 Agent 都能调"删库""转账"类工具），多租户场景还要在参数层注入 tenantId，不能让 Agent 自己传。
