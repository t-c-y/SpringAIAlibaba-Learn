订单查询 Agent：业务 Tool + 权限守卫

本课目标：把 Tool 落到真实业务，掌握"用户身份不由模型决定"这一生产必修课。

配套 Demo：demos/11-order-agent。参考卡：订单 Agent 快速参考。

你今天要获得的一个能力：能给 Spring AI Agent 的 Tool 方法安全地注入当前登录用户身份，彻底杜绝让大模型传 userId 导致的越权漏洞——你会用 ThreadLocal 把 HTTP 头里的 X-User-Id 一路透传到 @Tool 方法内部，并写出生产级的权限守卫。

三条边界

  1. 用户身份**不允许**模型自己填，必须来自服务端会话上下文（这里用 X-User-Id + ThreadLocal）。
  2. 工具方法内部**必须做权限校验**，Prompt 是软规则不算数。
  3. 写操作要走**人工确认 + 幂等 API**，本课只覆盖只读查询。

原理：带身份上下文的工具调用链

为什么绝不能把 userId 做成 @Tool 方法的入参？因为大模型在填 tool call 参数时完全不受你控制——它可能从用户的自然语言里"猜"一个 userId（"帮我查 u002 的订单"），可能 hallucinate 一个不存在的 ID，更糟的是攻击者可以在 prompt 里注入 userId=admin。模型传参 = 把身份认证交给了最不可信的一环。

正确做法是把身份从模型视野中彻底拿掉：HTTP 入口处从可信来源（JWT / Session / 网关签名头）解析用户 ID，写入 ThreadLocal，后面整条调用链——Controller → ChatClient → Tool 执行器 → @Tool 方法——都从 ThreadLocal 读取，请求结束在 finally 中清空。模型根本不知道 userId 存在，也就无从伪造。

┌──────────┐  X-User-Id: u001   ┌─────────────┐
│  Client  │ ─────────────────▶ │ Controller  │
└──────────┘  (网关/JWT校验后)    └──────┬──────┘
                                       │ UserContext.set("u001")
                                       ▼
                              ┌─────────────────┐
                              │   ChatClient    │
                              │  .prompt(...)   │
                              │  .tools(orderT) │──┐
                              └─────────────────┘  │
                                                   │ Spring AI 反射调用
                                                   ▼
                              ┌──────────────────────────┐
                              │  @Tool listMyOrders()    │
                              │  @Tool getOrderById(id)  │
                              │  @Tool cancelOrder(id)   │
                              │     │                    │
                              │     ▼                    │
                              │  UserContext.get()       │
                              │   → "u001" (可信)        │
                              │     │                    │
                              │     ▼                    │
                              │  订单服务按 uid 查询      │
                              │  + 二次权限校验           │
                              └──────────────────────────┘
                                       │
                              finally { UserContext.clear() }
                              (防止线程池复用串号)

这条链路的核心只有一句话：身份来自服务端上下文，不来自模型生成的参数。ThreadLocal 只是最简单的实现；在 WebFlux 反应式场景替换为 Reactor Context，在 gRPC 场景替换为 Context.Key，原理完全一致。

核心代码：OrderTools + UserContext

以下是订单 Agent 的完整工具类，三个方法都不接收 userId 参数，统一从 UserContext.currentUserId() 读取当前调用者身份。这是企业级"行级权限"的最小正确实现。

1) UserContext：ThreadLocal 持有者

public final class UserContext {
    private static final ThreadLocal<String> UID = new ThreadLocal<>();

    public static void set(String uid)  { UID.set(uid); }
    public static String currentUserId() {
        String uid = UID.get();
        if (uid == null) throw new IllegalStateException("未登录，X-User-Id 缺失");
        return uid;
    }
    public static void clear()          { UID.remove(); }
}

2) OrderTools：所有 @Tool 方法从上下文取 uid

@Component
public class OrderTools {
    private final OrderRepository orders;

    public OrderTools(OrderRepository orders) { this.orders = orders; }

    @Tool(description = "列出当前用户的所有订单")
    public List<OrderView> listMyOrders() {
        String uid = UserContext.currentUserId(); // 不从参数拿
        return orders.findByUserId(uid);          // 强制按 uid 过滤
    }

    @Tool(description = "按订单号查询单个订单，非本人订单返回无权限")
    public Map<String, Object> getOrder(String orderId) {
        String uid = UserContext.currentUserId();
        Order o = orders.findById(orderId).orElse(null);
        if (o == null)            return Map.of("found", false);
        if (!o.getUserId().equals(uid)) {
            // 关键：即便 orderId 存在，非本人也返回"查无此单"，
            // 避免攻击者枚举订单号探测订单是否存在
            return Map.of("found", false, "hint", "订单不存在或无权限");
        }
        return Map.of("found", true, "order", o);
    }

    @Tool(description = "取消订单。订单必须属于当前用户且状态为待支付。")
    public Map<String, Object> cancelOrder(String orderId) {
        String uid = UserContext.currentUserId();
        Order o = orders.findById(orderId).orElse(null);
        if (o == null || !o.getUserId().equals(uid))
            return Map.of("ok", false, "msg", "订单不存在");
        if (o.getStatus() != OrderStatus.PENDING_PAYMENT)
            return Map.of("ok", false, "msg", "当前状态不允许取消");
        o.setStatus(OrderStatus.CANCELLED);
        orders.save(o);
        return Map.of("ok", true, "msg", "已取消");
    }
}

3) Controller：入口绑定 + finally 清理

@RestController
@RequestMapping("/order")
public class OrderAgentController {
    private final ChatClient chatClient;

    public OrderAgentController(ChatClient.Builder builder,
                                OrderTools orderTools) {
        this.chatClient = builder.defaultTools(orderTools).build();
    }

    @PostMapping("/ask")
    public AskResp ask(@RequestBody AskReq req,
                       @RequestHeader("X-User-Id") String uid) {
        UserContext.set(uid);                       // 入口绑定
        try {
            String answer = chatClient.prompt(req.question()).call().content();
            return new AskResp(true, answer);
        } finally {
            UserContext.clear();                     // 必须在 finally 清
        }
    }
}

注意：生产环境 X-User-Id 不能直接信任前端传来的值，必须由网关/过滤器在验签或解析 JWT 后注入，或直接在 OncePerRequestFilter 里 set 到 ThreadLocal。

工具注册：默认工具 vs 按角色动态工具

上面示例用 defaultTools(orderTools) 把 OrderTools 的所有方法对所有用户开放。现实业务里你会遇到"客服能看所有订单、用户只能看自己、管理员能退款"这种按角色裁剪工具集的需求。两种选择：

defaultTools(...) — 默认工具

适用：登录后所有角色都能用的通用工具（如 listMyOrders、getOrder）。ChatClient 构建期绑定，每次调用都带上，性能最好。

写在：ChatClient.builder.defaultTools(...) 或 build() 之前。

.tools(...) — 按请求动态工具

适用：按角色/场景动态下发，例如管理员才挂 refundOrder，客服才挂 listAllOrders。在每次 prompt().tools(...).call() 时临时传入。

写在：chatClient.prompt(q).tools(toolsByRole(uid)).call()。

推荐实践：通用工具（用户自助查询）走 defaultTools，高危/特权工具走运行时动态 .tools(...)，并在动态工具内部仍然做 ThreadLocal + 二次角色校验——不要把"这个方法没挂到 tools()"当作安全边界，因为模型工具选择不是 100% 可预测，防御必须分层。

安全性对比：模型传 userId vs ThreadLocal 注入

维度让模型传 userIdThreadLocal 服务端注入

越权风险极高——prompt 注入即可换 uid低——uid 不经过模型

边界与常见坑（生产踩坑清单）：

  1. ThreadLocal 忘记 clear() 会串号。 Tomcat/Jetty 用线程池复用线程，如果某次请求异常没走到 finally 里的 clear()，下一个复用该线程的请求会读到上一个用户的 uid，导致跨用户数据泄露。必须用 try/finally 包裹，或注册一个 ServletRequestListener.requestDestroyed() 统一兜底清理。WebFlux 场景 ThreadLocal 直接失效（线程切换会丢），必须用 reactor.util.context.Context。
  2. ThreadLocal 不是唯一防线——工具内部必须做二次校验。 不要认为"uid 已经从 ThreadLocal 取了就安全"：工具可能被内部其他 Service 直接调用、可能被定时任务触发、可能被测试用例挂起，这时 ThreadLocal 里的值要么是 null 要么是上次残留。OrderRepository 查询层必须强制带 userId 条件（行级权限），最好加 @PreAuthorize 或 MyBatis 拦截器自动注入 uid 过滤，避免某条 SQL 漏写 where 条件。
  3. 写操作绝对不能让模型一句话就执行。 cancelOrder 看起来只是改个状态，但涉及退款、库存回补、履约中断。生产必须加幂等键（同一 orderId 多次 cancel 返回同样结果）、状态机校验、以及 HITL（Human-in-the-loop）确认——例如取消金额 > 500 元或已发货订单，要先把"将取消订单 SO-1001，退款 ¥1280，是否确认？"抛回给用户点按钮，不能模型直接调 Tool 提交。
  4. 订单读库走从库、写操作走主库。 列表查询和 getOrder 可以容忍秒级延迟，走从库扛流量；cancelOrder 这类写后立即读的操作必须走主库，否则刚 cancel 完下一句模型调 getOrder 查到从库可能还是"待发货"，模型就会胡编"取消失败"。
  5. @Tool 的 description 别把敏感表名/字段名暴露给模型。 你在 description 里写 "查询 t_order 表所有字段"，等于把数据库结构告诉 prompt 注入攻击者。描述要站在业务动作角度（"列出我的订单"），不要泄露内部实现。
