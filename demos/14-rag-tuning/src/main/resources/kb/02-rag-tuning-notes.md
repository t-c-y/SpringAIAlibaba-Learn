# RAG 参数调优心得

RAG 中常见调参维度：

1. Chunk Size（切片大小）
   - 太小：召回精细，但语义碎裂；模型看不到足够上下文时会拒答或胡编。
   - 太大：一个 chunk 包含多个话题，向量检索命中率下降，且回答易被无关内容干扰。
   - 经验值：400 ~ 800 token 是常见起点；技术文档偏中长，600 ~ 800 较稳；FAQ / QA 库偏短，200 ~ 400 更合适。

2. Chunk Overlap（切片重叠）
   - 保守值：chunkSize 的 20% ~ 30%。
   - 过大：存储和 embedding 成本都增加；过小：跨 chunk 的关键信息会被切碎。

3. topK（每次检索返回条数）
   - 太小：漏召回；太大：Prompt token 上升、成本高、且噪声可能挤掉有用片段。
   - 经验值：3 ~ 6，先从 4 开始试。

4. Similarity Threshold（相似度阈值）
   - 用来兜底：低于阈值的片段一律不给模型看，防止无关片段污染上下文。
   - 教学阶段可先关闭（设为 0），先看召回，再判断是否需要开启。

5. Metadata 过滤
   - 加 metadata：source、version、language、tenant。
   - 检索时使用 metadata 过滤（例如只查 version=latest），比"训练更好的向量模型"性价比高得多。

评估方法：为知识库准备一组"标准问题 + 标准答案"，逐组比较不同参数下的 hit@k、准确率、成本、延迟，形成一份"参数-指标"对照表，再选最优的一组进入生产。
